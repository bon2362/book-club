# Технический аудит Book Club — май 2026

**Скоуп:** структура БД, auth-цепочка, карта фич и границы Sheets↔БД.
**Стек:** Next.js 14 + NextAuth v5 + Neon Postgres (HTTP driver) + Drizzle ORM + Google Sheets + Resend + Vercel.
**Состояние:** 14 таблиц, 13 миграций, ~22 файла с прямым доступом к БД. Последние крупные изменения — введение `userIdentities` (миграция 0013) и `userActivityEvents` (0012), миграция при этом частично не завершена.

---

## Executive Summary

**Главный вывод.** Архитектура каталога книг и Sheets↔БД границы — здоровая. Проблемы сконцентрированы в auth-слое: после рефакторинга, который ввёл `userIdentities`, в БД параллельно живут три источника правды (`users` denormalized columns + `accounts` + `userIdentities`), пишутся в разное время с двумя разными нормализаторами провайдеров. Миграция работает через **lazy backfill при входе** (без explicit SQL-миграции), но `accounts` продолжает писаться параллельно с identity для google-юзеров — это не переходный костыль, а активная политика дублирования. Плюс — `@neondatabase/serverless` HTTP-driver **не поддерживает транзакции**, и весь identity-код пишет в несколько таблиц без atomicity.

**Топ-5 рисков:**

1. **`notification_queue` не чистится при удалении пользователя** — email удалённого юзера остаётся в очереди и уйдёт админу при следующем cron (GDPR-риск). `Critical`, усилия XS.
2. **HTTP-driver без транзакций** — `lib/user-identities.ts` явно ловит ошибку отсутствия транзакций и продолжает без atomicity. Любой частичный сбой оставляет orphan-данные. `Critical`, усилия M.
3. **`/api/auth/telegram/callback` без rate-limit + cleanup токенов раз в сутки** — атакующий может за 5-минутное окно валидной HMAC-подписи набить миллион строк в `telegram_preauth_tokens`. `Critical`, усилия M.
4. **`users.authProvider` + `users.telegramUsername` дублируют `userIdentities`** — два разных нормализатора, нет CHECK, перезаписываются на каждом входе. `High`, усилия M суммарно.
5. **`userActivityEvents` — строки никто не читает напрямую, но writer попутно обновляет `users.lastActivityAt`** — это поле использует админская колонка "Последняя активность". Не dead write, но избыточная детализация без читателя. См. H5. `Medium`, усилия XS–M.

Дальше — детали, граф auth-модели и план рефакторинга по шагам.

## 3. Граф auth-модели

```
                       ┌────────────────────────────┐
                       │  user (PK: id)             │  ← canonical user
                       │  UNIQUE email              │
                       │  + denormalized:           │
                       │    telegramUsername  ──┐   │
                       │    authProvider      ──┤   │  Дублируется в
                       │    lastSignInAt      ──┤   │  user_identities
                       │    lastActivityAt    ──┘   │  + user_activity_events
                       └──┬─────────────────────────┘
                          │ id (PK)
       ┌──────────────────┼──────────────────────┬───────────────────────┐
       ▼                  ▼                      ▼                       ▼
┌──────────────┐  ┌─────────────────┐    ┌──────────────────┐   ┌─────────────────────┐
│  account     │  │ user_identities │    │  session         │   │ user_activity_      │
│  PK (prov,   │  │ UNIQUE (prov,   │    │  PK token        │   │ events              │
│      provAcc)│  │    provAcc)     │    │  (НЕ ИСПОЛЬЗ.    │   │ UNIQUE dedupe_key   │
│              │  │                 │    │   при JWT strat) │   │ (WRITE-ONLY,        │
│  ← NextAuth  │  │ ← user-         │    │                  │   │  читателей нет)     │
│    adapter   │  │   identities.ts │    │                  │   │                     │
│  ← syncGoogle│  │                 │    │                  │   │                     │
└──────────────┘  └─────────────────┘    └──────────────────┘   └─────────────────────┘
       ▲                  ▲
       │                  │
       └──── DUBLE ───────┘
       Для provider='google' identity лежит в обеих таблицах.
       syncGoogleAccount явно пишет копию в accounts при каждом identity-write.
```

**Конфликтные точки:**
1. `accounts` и `user_identities` для google — двойная запись.
2. `users.telegramUsername` и `user_identities.telegramUsername` — двойная запись.
3. `users.authProvider` — два разных нормализатора (`lib/auth.ts:32-34` свёртывает только `resend→email`; `lib/user-identities.ts:62-67` свёртывает все три категории). В БД оседают оба домена значений одновременно.
4. `users.lastSignInAt` — дубль `userActivityEvents WHERE type='sign_in'`.

### Сценарии resolve identity

**Сценарий 1: первый логин через Google.** NextAuth adapter → `getUserByAccount` null → `getUserByEmail` null → `createUser` + `linkAccount`. `signIn` callback → update `users.authProvider='google'`, `linkIdentityToUser` пишет в `userIdentities`, `syncGoogleAccount` пытается перевставить `accounts` (no-op). Итог: 1 user, **identity для Google дублируется в `accounts` и `user_identities`**.

**Сценарий 2: тот же юзер через Email magic link на тот же адрес.** Adapter `getUserByEmail` находит существующего. `signIn` callback пишет `users.authProvider='email'` — **затирает 'google'**. `linkIdentityToUser('email', alice@gmail.com)` создаёт второй row в `userIdentities`. Итог: 1 user, 1 account (google), 2 identities, но `users.authProvider` потерял информацию про первичную регистрацию.

**Сценарий 3: тот же человек заходит через Telegram (Telegram email не отдаёт).** `findIdentityUserId` null, `findGoogleAccountUserId` null (rejects non-google), `canLinkByEmail` false (Telegram не отдаёт email). Создаётся **новый user** с синтетическим email `telegram:<id>@telegram.user`. Связать с существующим Google-аккаунтом по email — невозможно архитектурно (фундаментальное ограничение Telegram Login Widget).

## 4. Находки

### Critical

#### C1. `notification_queue` не очищается при удалении пользователя (GDPR-риск)
**Где:** `app/api/admin/delete-user/route.ts:24`, `app/api/user/route.ts:17`, `lib/db/schema.ts:156-168`
**Что:** `notification_queue` хранит `user_email` и `added_books` как сырые данные **без FK** на `users`. `delete-user` просто `DELETE FROM users` каскадно вычищает связанные таблицы, но `notification_queue` каскадом не цепляется. Pending-нотификации удалённого пользователя при следующем cron уйдут админу с его email. При этом тестовый эндпоинт `app/api/test/session/route.ts:60` явно делает `DELETE FROM notification_queue WHERE userEmail = email` — то есть про проблему знают, но только в test-режиме.
**Предложение:** перед `DELETE FROM users` явно чистить `notification_queue` по email. Альтернатива — добавить колонку `user_id` с FK CASCADE.
**Усилия:** XS

#### C2. `neon-http` driver не поддерживает транзакции — identity-операции пишут в несколько таблиц без atomicity
**Где:** `lib/user-identities.ts:44-60,247-321`, `lib/signup-books.ts:60-65`
**Что:** Drizzle с `@neondatabase/serverless` HTTP-driver не поддерживает настоящие транзакции — есть только batch-режим. Код в `user-identities.ts` явно ловит ошибку `"No transactions support in neon-http driver"` и продолжает без транзакции. Это значит:
- `resolveOrCreateUserFromIdentity`: создаётся `users.row` → падает на `userIdentities` insert → orphan-user с синтетическим email, который при следующей попытке войти не найдётся по identity (identity не записалась), а UNIQUE на `users.email` блокирует другую попытку.
- `upsertSignup` (`signup_books`): параллельные POST `/api/signup` от двух вкладок одного юзера → лост-write при гонке (запрос A удалил, B удалил пусто, A вставил [X,Y], B вставил [Z]).
**Предложение:** переключить `db` connection с `drizzle-orm/neon-http` на `drizzle-orm/neon-serverless` (WebSocket-driver — поддерживает транзакции). Это разблокирует все остальные правки и закрывает класс багов "частичная запись".
**Усилия:** M

#### C3. `/api/auth/telegram/callback` без rate-limit + cleanup токенов раз в сутки = flooding-вектор
**Где:** `app/api/auth/telegram/callback/route.ts:5-33`, `vercel.json` (cron `0 3 * * *`)
**Что:** Эндпоинт принимает Telegram-параметры из query string, проверяет HMAC (auth_date в окне 5 мин — это защищено хэшем, replay невозможен). Но атакующий, единожды видевший валидную HMAC-подпись, может в течение 5-минутного окна заспамить эндпоинт огромным числом запросов от того же `id` → каждый запрос создаёт row в `telegram_preauth_tokens`. Cleanup только раз в сутки → миллионы строк успеют накопиться, плюс `DELETE FROM telegram_preauth_tokens WHERE expires_at < now()` без `LIMIT` может выполняться на проде минутами.
**Предложение:**
- Rate limit (IP+telegram_id) 10/мин на эндпоинт.
- При INSERT нового pre-auth токена удалять прошлые токены того же `user_id` (one active token per user).
- Cleanup cron — добавить batched delete (`LIMIT 10000` в цикле) или чаще запускать (каждый час).
**Усилия:** M

#### C4. `linkIdentityToUser` падает с throw'ом и обрушивает весь signIn
**Где:** `lib/user-identities.ts:227-245` + `lib/auth.ts:171-179`
**Что:** В `signIn` callback вокруг `linkIdentityToUser` нет `try/catch`. Если `findGoogleAccountUserId` бросит "already linked to another user" (нестыковка после миграции, race, баг), весь auth-flow упадёт → юзер заблокирован до ручного вмешательства. Identity — вторичный кэш, не должна блокировать вход.
**Предложение:** обернуть `linkIdentityToUser` в `try/catch` с `console.error` + продолжить signIn.
**Усилия:** XS

### High

#### H1. `accounts` и `user_identities` параллельно — активное дублирование с lazy backfill
**Где:** `lib/db/schema.ts:37-66`, `lib/user-identities.ts:132-143,227-245,293-298`, `lib/auth.ts:171-200`

**Что:** Миграция 0013 добавила `user_identities`, но без SQL-backfill. Вместо этого выбран **lazy backfill в runtime**: при первом входе старого юзера identity-row создаётся через fallback-цепочку в `resolveOrCreateUserFromIdentity` (`user-identities.ts:293-298`): existing identity → `findGoogleAccountUserId` (через старую `accounts`) → email match → `findLegacyTelegramUserId` (через старые `id='telegram:NNN'`). `signIn` callback в `auth.ts:171-200` явно вызывает `linkIdentityToUser` для google и email — это создаёт identity для старых юзеров на лету.

Это работает. Но при этом:
1. **`accounts` продолжает писаться** на каждом identity-write google-юзера через `syncGoogleAccount` (`user-identities.ts:227-245`) — это не переходный костыль, а активная политика. Для google identity всегда лежит **в обеих таблицах одновременно**.
2. **Нет explicit SQL backfill** — старые юзеры, которые не входили после деплоя 0013, всё ещё без identity-row. Если кто-то решит удалить fallback-логику раньше, чем все юзеры успели зайти — они сломаются.
3. **Три fallback-пути висят в hot-path** (`findGoogleAccountUserId`, `findUserIdByEmail`, `findLegacyTelegramUserId`) — лишние SELECT'ы при каждом resolve.

NextAuth с `session: { strategy: 'jwt' }` адаптеру `accounts` фактически не нужна (только для Resend verifyEmail flow — там она используется для linking при первом OAuth).

**Предложение:**
1. Сделать explicit SQL backfill в новой миграции (`INSERT INTO user_identities SELECT ... FROM accounts WHERE provider='google'`) — закрыть hole для неактивных юзеров.
2. После backfill — удалить `syncGoogleAccount` (двойная запись больше не нужна).
3. Через 1-2 месяца, когда уверенность по логам, что все активные юзеры мигрировали — удалить fallback'и `findGoogleAccountUserId` и `findLegacyTelegramUserId` из `resolveOrCreateUserFromIdentity`. Оставить только `findIdentityUserId` + `findUserIdByEmail`.
4. Решить судьбу `accounts`: оставить только для Resend verifyEmail flow с явной пометкой "не source of truth", или кастомизировать DrizzleAdapter чтобы не писать туда.

**Усилия:** M (backfill + удаление syncGoogleAccount — S; cleanup fallback'ов — отдельный шаг через 1-2 месяца, XS)

#### H2. `users.authProvider` — два нормализатора, нет CHECK, перезаписывается на каждом входе
**Где:** `lib/db/schema.ts:14-15`, `lib/auth.ts:32-34`, `lib/user-identities.ts:62-67`
**Что:** Колонка text без enum. `normalizeAuthProvider` (auth.ts) свёртывает только `resend → email`; `normalizeIdentityProvider` (user-identities.ts) свёртывает все три (`google-one-tap → google`, `telegram-preauth → telegram`). В одной БД одновременно: `google`, `google-one-tap`, `email`, `telegram`, `telegram-preauth`. Группировка по auth_provider даёт мусор. Плюс — на каждом входе перезаписывается значением последнего провайдера, теряется история ("как был зарегистрирован изначально"). Источник правды для "каким провайдером юзер заходил" — `userIdentities` по `lastSeenAt`.
**Предложение:** удалить колонку `users.authProvider` целиком. В админке и `/api/me` брать из `userIdentities ORDER BY lastSeenAt DESC LIMIT 1`. Минимум — единый нормализатор + CHECK.
**Усилия:** M

#### H3. `users.telegramUsername` дублирует `user_identities.telegramUsername`
**Где:** `lib/db/schema.ts:13,43`, `lib/auth.ts:161`, `lib/user-identities.ts:182-192,302-313`
**Что:** Оба поля обновляются параллельно (`updateUserCache` пишет в users, `upsertIdentity` — в identities) в разных кодовых путях. На чтение используется только `users.telegramUsername` (`lib/admin-users.ts:78,120,140`) — `user_identities.telegramUsername` — мёртвый writer. При расхождении (если юзер сменит username между запросами) — баг в админке.
**Предложение:** убрать `telegramUsername` из `user_identities` (или унести в `metadata`), оставить только в `users`. Или наоборот — убрать из `users` и читать через JOIN.
**Усилия:** S

#### H4. `users.lastSignInAt` — дубль `userActivityEvents WHERE type='sign_in'`
**Где:** `lib/db/schema.ts:16-17`, `lib/auth.ts:160`, `lib/user-identities.ts:185,310`
**Что:** После того как появились events, `lastSignInAt` — отдельная денормализация без читателя кроме `/api/me/route.ts:23`, который мог бы взять `lastActivityAt`. Колонка пишется в трёх местах, читается в одном. `lastActivityAt` оправдана как кэш для сортировки в админке (без джойна).
**Предложение:** удалить `users.lastSignInAt`. В `/api/me` отдавать `lastActivityAt`.
**Усилия:** S

#### H5. `user_activity_events` — детальные строки не читаются, но writer обновляет `users.lastActivityAt`
**Где:** `lib/user-activity.ts:49-66,84` (writer + UPDATE на `users.lastActivityAt`); прямых SELECT'ов из таблицы нет (`grep -rn from(userActivityEvents)` пусто).
**Что:** На каждый `sign_in / profile_submitted / books_selected / priorities_updated / submission_created / feedback_created` идёт INSERT в `user_activity_events` **плюс** в той же операции `UPDATE users SET lastActivityAt = greatest(lastActivityAt, occurredAt)`. Админка показывает колонку "Последняя активность" через `users.lastActivityAt` (`components/nd/AdminPanel.tsx:870`) — то есть денормализованный кэш, который поддерживает этот writer. Поля самих событий (`type`, `source`, `metadata`, `dedupeKey`) при этом никто не читает.
**Почему это High, а не "выкинуть":** удалить таблицу нельзя без замены — иначе перестанет обновляться `lastActivityAt`. Сейчас это не dead write, а **избыточная детализация поверх работающего кэша**.
**Предложение:** три варианта по убыванию работы —
- (a) Подключить читателя: timeline активности в `/admin/users/[id]` (что было задумано миграцией 0012). Тогда таблица обретает полный смысл. Усилия: M.
- (b) **Оставить как есть** + добавить комментарий в `lib/user-activity.ts`, что writer параллельно поддерживает `users.lastActivityAt`. Дешевле всего, потери нет. Усилия: XS.
- (c) Упростить: удалить таблицу `user_activity_events`, оставить только UPDATE `users.lastActivityAt` в той же функции. Теряем `type/source/metadata` (могут пригодиться для аналитики), но админка продолжит работать. Усилия: S.

Рекомендация: **вариант (b)** до момента, когда появится бизнес-задача типа "таймлайн активности юзера" или "сколько юзеров активны за неделю". Накладные расходы на INSERT минимальны, а сырые события дешевле собирать заранее, чем восстанавливать задним числом.

#### H6. Email magic link может создать pre-existing user с чужим email
**Где:** `lib/auth.ts:151-202`, `lib/user-identities.ts:100-105`
**Что:** Если атакующий запросит magic link на `victim@gmail.com`, который ещё не в системе, и жертва из любопытства кликнет (а не атакующий) — NextAuth создаст user. Потом жертва зайдёт через Google → adapter `getUserByEmail` найдёт этого user → присоединит Google account. Атакующий уже не получит сессию (ссылку открыла жертва), но если жертва **никогда** не зайдёт, а атакующий — да... Сейчас защиты нет, потому что Email-провайдер по дизайну требует доступ к почте — атакующий ссылку не увидит. Риск низкий, но архитектурно: нет различия "запросил-ссылку" от "запросил-и-перешёл". `email_verified` для Google Workspace может быть false — код корректно проверяет, но это узкое место.
**Предложение:** добавить unit-test на создание user только при первом успешном callback (не при request). Сейчас это, кажется, и так работает (`verificationRequest=true` возвращает true рано, без создания user) — но тест зафиксирует контракт.
**Усилия:** S

#### H7. `signup_books` и `book_priorities` — две таблицы вместо одной, связь поддерживается вручную
**Где:** `lib/db/schema.ts:118-133`, `app/api/signup/route.ts:35-53`, `app/api/priorities/route.ts`, `app/api/admin/rename-book/route.ts:21-27`
**Что:** Две таблицы с одинаковым PK `(user_id, book_name)` и неявной связью "приоритет = ранг внутри своих записей":
- `app/api/signup/route.ts:35-53` вручную удаляет приоритеты для снятых книг. Любой новый код, меняющий `signup_books`, обязан помнить про эту чистку — иначе осиротевшие приоритеты.
- `app/api/priorities/route.ts` не проверяет, что юзер записан на эти книги. Структурно возможен приоритет без подписи (UI этого не позволяет, но БД не enforce).
- `app/api/admin/rename-book/route.ts:21-27` делает два отдельных UPDATE в обе таблицы. Без транзакций (см. C2) частичный сбой → рассинхрон.

С учётом C2 (нет транзакций в HTTP-driver) каждая мульти-таблица операция здесь — потенциальный race / partial write. Объединение устраняет класс багов структурно, а не "не забудьте обновить обе".

**Предложение:** объединить в одну таблицу `signup_books`:
```
userId | bookName | signedAt | rank (nullable) | rankUpdatedAt (nullable)
```
`rank=null` — юзер записан, но не ранжировал. Обычное состояние сразу после записи. Семантика "запись" и "ранжирование" остаются разделены через `signedAt` vs `rankUpdatedAt`.

**Эффекты:**
- Невозможен осиротевший приоритет — структурное свойство схемы.
- `CASCADE` от удаления записи автоматически удаляет приоритет — убрать ручную чистку в `signup/route.ts:35-53`.
- `rename-book` — один UPDATE.
- Админка `lib/admin-users.ts:161-164` — один SELECT вместо двух (счас тянутся в память отдельно).

**Миграционная проверка** (выполнить перед миграцией):
```sql
SELECT bp.user_id, bp.book_name
FROM book_priorities bp
LEFT JOIN signup_books sb USING (user_id, book_name)
WHERE sb.user_id IS NULL;
```
Если результат пуст — миграция простой LEFT JOIN'ом. Если есть строки — продуктово решить: создать запись в signup_books или дропнуть приоритет.

**Файлы под правки:** `lib/db/schema.ts`, новая миграция `0014_merge_priorities.sql`, `lib/signup-books.ts` (`upsertSignup` добавляет `rank: null`), `app/api/priorities/route.ts` (INSERT → UPDATE, убрать `notInArray`-чистку), `app/api/signup/route.ts:35-53` (удалить блок чистки), `app/api/admin/rename-book/route.ts` (один UPDATE), `lib/admin-users.ts:161-164`, тесты.

**Усилия:** S (день работы с тестами).

## 5. Рекомендуемый план рефакторинга

Логика порядка: сначала data-loss/privacy риски (Шаг 1), потом инфраструктурный вопрос транзакций (Шаг 2) — это разблокирует чистую нормализацию (Шаги 3–5). Параллельно мелкие индексы и checks (Шаги 6–7). Долгие исследовательские шаги (8 — нужен ли таймлайн activity events?) — в последнюю очередь.

**Шаг 1. Закрыть GDPR и блокирующие баги.**
- C1: чистить `notification_queue` при `DELETE FROM users` в `/api/admin/delete-user` и `/api/user`. (XS)
- C4: `try/catch` вокруг `linkIdentityToUser` в signIn callback. (XS)
- M6: hard-check `VERCEL_ENV === 'production'` в `/api/test/*`. (XS)

**Шаг 2. Решить вопрос с транзакциями.**
- C2: переключить connection на `drizzle-orm/neon-serverless` (WebSocket-driver). Это разблокирует все остальные шаги, потому что многие правки требуют atomicity. (M)
- Альтернатива (если WebSocket не подходит для serverless cost): advisory-lock per-user в hot-paths.

**Шаг 3. Закрыть Telegram-flooding.**
- C3: rate-limit на `/api/auth/telegram/callback`, one-active-token per user, batched cleanup. (M)

**Шаг 4. Завершить identity-миграцию.**
- H1: backfill `accounts → user_identities` для google-юзеров; решить судьбу `accounts` (оставить только для Resend verifyEmail или удалить с кастомным адаптером); удалить `syncGoogleAccount`. (M)
- H2: удалить колонку `users.authProvider` целиком (читать из `userIdentities`). (M)
- H3: убрать `telegramUsername` из `userIdentities` или из `users` — выбрать одну сторону. (S)
- H4: удалить `users.lastSignInAt`, в `/api/me` использовать `lastActivityAt`. (S)

**Шаг 5. Объединить `signup_books` и `book_priorities`.**
- H7: одна таблица с `rank nullable` вместо двух. Перед миграцией прогнать проверочный SQL (см. H7) на наличие осиротевших приоритетов. Лучше делать после Шага 2 (транзакции) — миграционный INSERT/DROP безопаснее под настоящими транзакциями, но в крайнем случае можно и без. Закрывает класс багов "рассинхрон между таблицами при rename/delete" + убирает 20 строк ручной чистки в `/api/signup`. (S)

**Шаг 6. Решить судьбу `user_activity_events`.**
- H5: writer одновременно обновляет `users.lastActivityAt` для админки, поэтому "просто отключить" нельзя. Рекомендация: вариант (b) — оставить + комментарий (XS). Если появится продуктовая задача типа timeline активности — реализовать вариант (a), читатель в `/admin/users/[id]` (M).

**Шаг 7. Мелкие индексы и checks.**
- M4: partial index `notification_queue (processing_at) WHERE sent_at IS NULL`. (XS)
- M7: сбросить `priorities_set` в `/api/signup`. (XS)
- M8: индексы на `book_name` (после Шага 5 — только в объединённой таблице). (XS)
- M9: `force-dynamic` на auth-маршруты. (XS)
- L2: CHECK constraints для text-enum. (XS)

**Шаг 8. Очистка legacy.**
- M1: `DROP TABLE session` (если решено не возвращаться к Database strategy). (S)
- M3: удалить `ensureIntroTable` runtime bootstrap, seed через миграцию. (S)
- L4: удалить legacy `telegram` Credentials provider. (XS)
- L3, L5: проверить потребителей `/api/books` и `/api/admin/route.ts`, удалить или выровнять. (S)

**Шаг 9. Полировка типов.**
- M2: `text → jsonb` для `metadata` и `languages`. (S/M)
