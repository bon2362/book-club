# book-club - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for book-club, decomposing the requirements from the PRD and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: Авторизованный пользователь может открыть форму заявки на книгу
FR2: Авторизованный пользователь может заполнить заявку со следующими полями: Название (обязательное), Тема (выбор из тем каталога), Писатель (обязательное), Число страниц, Дата издания, Ссылка на текст, Описание, Ссылка на обложку, Почему стоит прочитать (обязательное)
FR2a: Система предоставляет список доступных тем для выбора (на основе тем из каталога Google Sheets)
FR3: Авторизованный пользователь может отправить заявку
FR4: Система отображает подтверждение после успешной отправки заявки
FR5: Система сохраняет заявку со статусом `pending` и привязывает её к пользователю
FR6: Псевдо-карточка "Предложить книгу" отображается первой в сетке каталога для всех пользователей
FR7: Кнопка "Предложить книгу" отображается в хедере для всех пользователей
FR8: При клике незалогиненного пользователя на любой из входов система показывает попап авторизации
FR9: После успешной авторизации система автоматически открывает форму заявки
FR10: Авторизованный пользователь открывает форму напрямую — без попапа авторизации
FR11: Система отправляет пользователю email при одобрении его заявки
FR12: Система отправляет пользователю email при отклонении его заявки
FR13: Одобренные пользовательские книги отображаются в общем каталоге наравне с книгами администратора
FR14: Любой пользователь (включая незалогиненного) может просматривать каталог
FR15: Администратор может просматривать список всех заявок с их статусами
FR16: Администратор может просматривать все поля заявки (включая "Почему стоит прочитать?" и email автора)
FR17: Администратор может редактировать поля заявки перед принятием решения
FR18: Администратор может одобрить заявку — книга становится видна в каталоге
FR19: Администратор может отклонить заявку

### NonFunctional Requirements

NFR1: (Mobile UX) Псевдо-карточка, кнопка в хедере и форма заявки корректно отображаются и используются на мобильных устройствах
NFR2: (Mobile UX) Touch-цели (кнопки, поля, ссылки) имеют достаточный размер для нажатия пальцем
NFR3: (Mobile UX) Форма не требует горизонтальной прокрутки; поля вызывают подходящий тип мобильной клавиатуры (текст, число, URL, дата)
NFR4: (Mobile UX) Попап авторизации и форма заявки не перекрываются системными элементами (адресная строка, клавиатура)
NFR5: (Security) Форма заявки доступна только аутентифицированным пользователям — проверка сессии на сервере
NFR6: (Security) Admin-панель доступна только пользователям с флагом `isAdmin` — проверка на уровне API-маршрутов
NFR7: (Security) Данные заявок не раскрываются через API неавторизованным пользователям
NFR8: (Integration) Resend — сбой доставки email не блокирует операцию администратора
NFR9: (Integration) Neon Postgres / Drizzle — новая таблица `book_submissions` добавляется через `drizzle-kit` миграцию
NFR10: (Integration) Google Sheets — список тем берётся из уже загруженного кэша (`fetchBooks()`) без дополнительных запросов к Sheets API

### Additional Requirements

- Новая таблица `book_submissions` в `lib/db/schema.ts` с колонками: id (uuid, PK), user_id (text, FK → users.id), title, topic, author, pages, published_date, text_url, description, cover_url, why_read, status ('pending'|'approved'|'rejected'), created_at, updated_at
- Миграция через `drizzle-kit`; индекс на `status` для фильтрации в admin-панели
- Новый React Context `SubmitBookIntentContext` на уровне `app/new-design/layout.tsx` для координации auth state → UI state (открытие формы после авторизации)
- Расширить `/api/books` — объединить данные Google Sheets с одобренными заявками из БД в единый ответ
- Новые API маршруты: `POST /api/submissions` (auth), `GET /api/admin/submissions` (isAdmin), `PATCH /api/admin/submissions/[id]` (isAdmin)
- Email-шаблоны в отдельном файле `lib/email-templates/submission-status.ts` — функции `approvedEmail(bookTitle)` и `rejectedEmail(bookTitle)`
- Порядок реализации: 1) DB schema + миграция → 2) email-шаблоны → 3) API маршруты → 4) расширить /api/books → 5) SubmitBookIntentContext → 6) UI компоненты → 7) Admin секция

## Epic 1: Участник предлагает книгу

Участники клуба могут предложить книгу через форму заявки — из хедера или псевдо-карточки в каталоге. Незалогиненный пользователь проходит авторизацию и сразу попадает в форму. Заявка сохраняется в БД со статусом `pending`, пользователь получает подтверждение.

### Story 1.1: База данных и API подачи заявки

As an authenticated user,
I want to submit a book proposal via API,
So that my submission is saved for admin review.

**Acceptance Criteria:**

**Given** аутентифицированный пользователь отправляет POST на `/api/submissions` с валидными обязательными полями (title, author, whyRead)
**When** запрос обрабатывается
**Then** заявка сохраняется в таблице `book_submissions` со статусом `pending` и привязывается к user ID из сессии

**Given** запрос на POST `/api/submissions` без валидной сессии
**When** запрос обрабатывается
**Then** API возвращает 401 Unauthorized

**Given** запрос без хотя бы одного обязательного поля (title, author или whyRead)
**When** запрос обрабатывается
**Then** API возвращает 400 с сообщением об ошибке валидации

**Given** валидный запрос с незаполненными необязательными полями
**When** запрос обрабатывается
**Then** заявка сохраняется с `null` для необязательных полей

### Story 1.2: Форма заявки для авторизованного пользователя

As an authenticated user,
I want to fill and submit a book proposal form with all required fields,
So that I can propose a book for the club catalog and receive confirmation.

**Acceptance Criteria:**

**Given** авторизованный пользователь открывает форму
**When** форма отображается
**Then** форма содержит все 9 полей: Название (required), Тема (dropdown), Писатель (required), Число страниц, Дата издания, Ссылка на текст, Описание, Ссылка на обложку, Почему стоит прочитать (required)

**Given** дропдаун тем открыт
**When** пользователь видит варианты
**Then** список тем соответствует темам из каталога (из кэша Google Sheets, без дополнительных запросов к API Sheets)

**Given** авторизованный пользователь заполнил обязательные поля и нажал "Отправить"
**When** форма отправлена успешно
**Then** отображается подтверждение: "Заявка принята! Мы рассмотрим её в ближайшее время."

**Given** пользователь нажимает "Отправить" с незаполненными обязательными полями
**When** форма валидируется
**Then** форма показывает ошибки валидации для каждого пустого обязательного поля

**Given** форма открыта на мобильном устройстве
**When** пользователь взаимодействует с ней
**Then** форма не требует горизонтальной прокрутки и каждое поле вызывает подходящий тип клавиатуры (текст, число, URL, дата)

### Story 1.3: Точки входа и авторизация незалогиненных

As any user (authenticated or not),
I want to access the book submission form from the header and the catalog grid,
So that I can easily discover and initiate a book submission from anywhere on the site.

**Acceptance Criteria:**

**Given** любой пользователь открывает страницу каталога
**When** страница загружается
**Then** псевдо-карточка "Предложить книгу" отображается первой в сетке книг

**Given** любой пользователь видит хедер
**When** страница загружается
**Then** кнопка "Предложить книгу" видна в хедере

**Given** авторизованный пользователь кликает на псевдо-карточку или кнопку в хедере
**When** клик происходит
**Then** форма заявки открывается напрямую — без попапа авторизации

**Given** неавторизованный пользователь кликает на псевдо-карточку или кнопку в хедере
**When** клик происходит
**Then** отображается попап авторизации

**Given** неавторизованный пользователь завершил вход через попап, открытый по клику на точку входа
**When** авторизация успешна
**Then** форма заявки открывается автоматически — без повторного клика пользователя

## Epic 2: Администратор модерирует заявки

Администратор может просматривать все заявки, редактировать поля, одобрять или отклонять. При смене статуса автор заявки получает email-уведомление.

### Story 2.1: Email-шаблоны уведомлений о статусе заявки

As a system,
I want to send email notifications when a submission status changes,
So that users are informed about the outcome of their proposals.

**Acceptance Criteria:**

**Given** функция `approvedEmail(bookTitle)` вызывается
**When** она выполняется
**Then** возвращается объект `{ subject, html }` с темой письма и HTML-телом, содержащим название книги и сообщение об одобрении

**Given** функция `rejectedEmail(bookTitle)` вызывается
**When** она выполняется
**Then** возвращается объект `{ subject, html }` с темой письма и HTML-телом, содержащим название книги и нейтральное сообщение об отклонении

**Given** шаблоны подключены в проекте
**When** импортируются из `lib/email-templates/submission-status.ts`
**Then** экспортируются как named exports (не default export), совместимые с Resend API

### Story 2.2: API модерации заявок

As an admin,
I want to access a list of submissions and update their status via API,
So that I can manage book proposals programmatically.

**Acceptance Criteria:**

**Given** запрос GET на `/api/admin/submissions` от пользователя с флагом `isAdmin`
**When** запрос обрабатывается
**Then** возвращается список всех заявок со всеми полями (включая email автора) и их статусами

**Given** запрос GET без флага `isAdmin`
**When** запрос обрабатывается
**Then** API возвращает 403 Forbidden

**Given** PATCH запрос на `/api/admin/submissions/[id]` с новым статусом (`approved` или `rejected`) от администратора
**When** запрос обрабатывается
**Then** статус заявки обновляется в БД, и вызывается отправка email-уведомления через Resend

**Given** Resend возвращает ошибку при отправке email
**When** ошибка возникает
**Then** операция смены статуса всё равно считается успешной; ошибка логируется в консоль

**Given** PATCH запрос включает изменения полей заявки (не только статус)
**When** запрос обрабатывается
**Then** поля заявки обновляются вместе со статусом

**Given** запрос PATCH без флага `isAdmin`
**When** запрос обрабатывается
**Then** API возвращает 403 Forbidden

### Story 2.3: Admin-панель модерации в UI

As an admin,
I want to see and manage all book submissions in the admin panel,
So that I can review, edit, approve or reject proposals without using the API directly.

**Acceptance Criteria:**

**Given** администратор открывает admin-страницу
**When** страница загружается
**Then** отображается секция "Заявки" со списком всех заявок и их статусами (`pending`, `approved`, `rejected`)

**Given** администратор кликает на заявку в списке
**When** открывается детальный вид
**Then** отображаются все поля заявки, включая "Почему стоит прочитать?" и email автора

**Given** администратор редактирует поля заявки (например, исправляет опечатку)
**When** изменения сохраняются
**Then** обновлённые данные отражаются в БД

**Given** администратор нажимает кнопку "Одобрить"
**When** действие выполняется
**Then** статус заявки меняется на `approved`, автор получает email-уведомление, UI обновляется

**Given** администратор нажимает кнопку "Отклонить"
**When** действие выполняется
**Then** статус заявки меняется на `rejected`, автор получает email-уведомление, UI обновляется

## Epic 3: Одобренные книги в общем каталоге

Одобренные заявки появляются в каталоге наравне с книгами администратора (из Google Sheets). Каталог остаётся публичным — просматривать могут все.

### Story 3.1: Объединённый каталог из двух источников

As any user (authenticated or not),
I want to see approved user-submitted books alongside admin-curated books in the catalog,
So that the catalog reflects the collective expertise of the club's members.

**Acceptance Criteria:**

**Given** GET запрос на `/api/books`
**When** запрос обрабатывается
**Then** ответ содержит объединённый список: книги из Google Sheets + заявки со статусом `approved` из БД

**Given** книга из заявки одобрена (статус `approved`)
**When** пользователь открывает каталог
**Then** одобренная книга видна в каталоге наравне с книгами администратора

**Given** заявка имеет статус `pending` или `rejected`
**When** пользователь открывает каталог
**Then** такая книга не отображается в каталоге

**Given** незалогиненный пользователь открывает каталог
**When** страница загружается
**Then** каталог доступен без авторизации — книги видны всем

**Given** в БД нет одобренных заявок
**When** запрос на `/api/books` обрабатывается
**Then** возвращаются только книги из Google Sheets — без ошибок
