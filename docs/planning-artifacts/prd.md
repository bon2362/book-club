# Product Requirements Document - book-club

**Author:** Evgenii
**Date:** 2026-03-15

## Executive Summary

Сайт книжного клуба "Долгое наступление" расширяется от статичного каталога (только Google Sheets) к живой платформе, где участники предлагают книги сами. Целевая аудитория — члены клуба, авторизованные через Google OAuth или magic-link. Каталог перестаёт быть только выбором администратора и начинает отражать коллективную экспертизу участников.

Новый функционал: залогиненный пользователь заполняет форму заявки, заявка проходит модерацию, при смене статуса — email-уведомление. Одобренные книги отображаются в общем каталоге наравне с книгами администратора.

### What Makes This Special

Поле "Почему стоит прочитать?" создаёт органические рекомендации от реальных участников клуба. Каждая одобренная книга несёт личную рекомендацию того, кто её предложил. Участники становятся соавторами каталога, а не только его потребителями.

## Success Criteria

### User Success

- После отправки формы пользователь видит подтверждение: заявка принята и находится на рассмотрении
- Пользователь получает email при смене статуса заявки (одобрена / отклонена)
- Форма доступна из двух мест на главной без лишних шагов (хедер + псевдо-карточка в сетке)
- Одобренная книга появляется в каталоге — пользователь видит результат своего вклада

### Technical Success

- Заявки сохраняются в Neon Postgres без потерь
- Email-уведомления доставляются через Resend при каждом изменении статуса
- Форма доступна только залогиненным пользователям
- Данные заявок корректно отделены от книг администратора (Google Sheets)
- Admin-панель позволяет просматривать, редактировать, одобрять и отклонять заявки

### Measurable Outcomes

- Форма не теряет данные; нет неперехваченных ошибок при отправке
- Email уходит в течение нескольких секунд после смены статуса

## Product Scope

### MVP Strategy

Problem-solving MVP — минимальный набор, при котором фича полезна с первого дня: участник предлагает книгу, администратор принимает решение, участник узнаёт о нём. Один разработчик, существующий стек (Next.js, Neon Postgres, Drizzle, Resend, NextAuth).

### MVP Feature Set (Phase 1)

- Форма заявки с 9 полями (см. FR2); обязательные: название, писатель, "Почему стоит прочитать?"
- Хранение заявок в таблице `book_submissions` (Neon Postgres / Drizzle)
- Статусная модель: `pending → approved / rejected`
- Email-уведомление при смене статуса (Resend)
- Подтверждение в UI после отправки
- Admin-панель: список заявок, просмотр всех полей, редактирование, approve / reject
- Кнопка в хедере + псевдо-карточка #1 в сетке → открывают форму или попап авторизации
- После авторизации через попап — автоматически открывается форма заявки
- Одобренные книги отображаются в общем каталоге

### Phase 2 (Growth)

- Загрузка обложки файлом (сейчас — только URL)
- Страница "Мои заявки" для пользователя (история + статусы)
- Комментарий администратора при отклонении (объяснение причины)

### Phase 3 (Expansion)

- Голосование участников за предложенные книги
- Автозаполнение данных книги по ISBN/названию

### Risk Mitigation

**Technical:** Весь стек уже используется. Нетривиальный момент — передача состояния "открыть форму" после авторизации через попап; решается через URL-параметр или React context.

**Resource:** Фича независима от остальных частей системы — может быть отложена без ущерба для существующей функциональности.

## User Journeys

### Journey 1: Участник предлагает книгу (happy path)

Маша — участница клуба, недавно прочитала "Клуб убийств по четвергам" и хочет предложить её для общего списка. Заходит на сайт, видит первую карточку-приглашение "Предложить книгу" в сетке. Кликает (она уже залогинена). Заполняет форму, пишет в поле "Почему стоит прочитать?": "Это детектив, в котором смеёшься вслух — редкость для жанра". Нажимает "Отправить" — видит: *"Заявка принята! Мы рассмотрим её в ближайшее время."* Через день получает email: заявка одобрена. Открывает сайт — книга в каталоге.

**Раскрывает:** форма заявки, подтверждение в UI, хранение в БД, email при одобрении, одобренные книги в каталоге.

---

### Journey 2: Незалогиненный посетитель хочет предложить книгу

Игорь зашёл на сайт впервые. Видит псевдо-карточку "Предложить свою книгу". Кликает — появляется попап авторизации. Авторизуется через Google. Попап закрывается — форма добавления книги открывается сразу, без потери контекста. Дальше — happy path.

**Раскрывает:** защита формы (только auth), попап авторизации при клике незалогиненного, автоматическое открытие формы после входа.

---

### Journey 3: Участник получает отказ

Петя предложил книгу, которую администратор уже планировал добавить из Sheets. Заявка отклонена. Петя получает email: *"Ваша заявка на книгу [название] не была одобрена."*

**Раскрывает:** email при отклонении, нейтральная формулировка уведомления.

---

### Journey 4: Администратор модерирует заявки

Администратор открывает admin-панель. Видит список заявок со статусом `pending`. Одну одобряет — книга появляется в каталоге, автор получает email. В другой правит опечатку в названии, затем одобряет. Третью отклоняет — дубликат.

**Раскрывает:** admin-панель со списком заявок, просмотр всех полей, редактирование перед одобрением, кнопки approve / reject.

## Web App Technical Context

Next.js 14 App Router, гибридный рендеринг. Сайт закрытый — индексация не нужна, аудитория фиксированная. Новый функционал вписывается в существующую архитектуру без новых зависимостей.

- Поддержка браузеров: современные (Chrome, Firefox, Safari, Edge)
- Адаптивность: все точки входа (псевдо-карточка, кнопка в хедере), попап авторизации и форма заявки должны удобно работать на мобильных устройствах — достаточные размеры touch-целей, форма не требует горизонтальной прокрутки, поля корректно вызывают мобильную клавиатуру
- Доступность: базовый уровень — семантический HTML, клавиатурная навигация, ARIA для попапа формы
- Real-time не требуется — статусы обновляются при следующей загрузке

## Functional Requirements

### Подача заявки (Book Submission)

- **FR1:** Авторизованный пользователь может открыть форму заявки на книгу
- **FR2:** Авторизованный пользователь может заполнить заявку со следующими полями:
  - Название (на русском, если есть перевод) — **обязательное**
  - Тема (выбор из тем, представленных на сайте)
  - Писатель (имя на языке оригинала) — **обязательное**
  - Число страниц
  - Дата издания (предлагаемого к чтению)
  - Ссылка на текст (если есть)
  - Описание
  - Ссылка на обложку (если есть)
  - Почему стоит прочитать — **обязательное**
- **FR2a:** Система предоставляет список доступных тем для выбора (на основе тем из каталога)
- **FR3:** Авторизованный пользователь может отправить заявку
- **FR4:** Система отображает подтверждение после успешной отправки заявки
- **FR5:** Система сохраняет заявку со статусом `pending` и привязывает её к пользователю

### Доступ к форме (Entry Points)

- **FR6:** Псевдо-карточка "Предложить книгу" отображается первой в сетке каталога для всех пользователей
- **FR7:** Кнопка "Предложить книгу" отображается в хедере для всех пользователей
- **FR8:** При клике незалогиненного пользователя на любой из входов система показывает попап авторизации
- **FR9:** После успешной авторизации система автоматически открывает форму заявки
- **FR10:** Авторизованный пользователь открывает форму напрямую — без попапа авторизации

### Уведомления (Notifications)

- **FR11:** Система отправляет пользователю email при одобрении его заявки
- **FR12:** Система отправляет пользователю email при отклонении его заявки

### Каталог книг (Book Catalog)

- **FR13:** Одобренные пользовательские книги отображаются в общем каталоге наравне с книгами администратора
- **FR14:** Любой пользователь (включая незалогиненного) может просматривать каталог

### Модерация (Admin)

- **FR15:** Администратор может просматривать список всех заявок с их статусами
- **FR16:** Администратор может просматривать все поля заявки (включая "Почему стоит прочитать?" и email автора)
- **FR17:** Администратор может редактировать поля заявки перед принятием решения
- **FR18:** Администратор может одобрить заявку — книга становится видна в каталоге
- **FR19:** Администратор может отклонить заявку

## Non-Functional Requirements

### Mobile UX

- Псевдо-карточка, кнопка в хедере и форма заявки корректно отображаются и используются на мобильных устройствах
- Touch-цели (кнопки, поля, ссылки) имеют достаточный размер для нажатия пальцем
- Форма не требует горизонтальной прокрутки; поля вызывают подходящий тип мобильной клавиатуры (текст, число, URL, дата)
- Попап авторизации и форма заявки не перекрываются системными элементами (адресная строка, клавиатура)

### Security

- Форма заявки доступна только аутентифицированным пользователям (проверка сессии на сервере)
- Admin-панель доступна только пользователям с флагом `isAdmin` (проверка на уровне API-маршрутов)
- Данные заявок не раскрываются через API неавторизованным пользователям

### Integration

- **Resend:** email-уведомления при смене статуса заявки; сбой доставки не блокирует операцию администратора
- **Neon Postgres / Drizzle:** новая таблица `book_submissions` в существующей схеме; миграция через `drizzle-kit`
- **Google Sheets:** список тем берётся из уже загруженного кэша каталога — без дополнительных запросов к Sheets
