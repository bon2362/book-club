# Разделение саммари по авторам на странице книги

**Дата:** 2026-06-28
**Статус:** утверждён дизайн, ожидает план реализации
**Область:** публичная страница чтения `/books/[bookSlug]/summaries`

## Проблема

Сейчас все опубликованные саммари книги от разных авторов склеены в одну
вертикальную простыню (`summaries.map` с `border-bottom` между статьями).
Читатель не понимает, где заканчивается текст одного автора и начинается
другого. Нужно явно разделить саммари по авторам.

## Решение

Переключатель авторов под шапкой книги: ряд пилюль (аватар + имя), **одно
саммари показывается за раз**. Выбранный автор отражён в URL через
`?author=<slug>` (SSR), что даёт расшаривание ссылкой и работающую кнопку
Back. В конце ряда — CTA «+ Написать своё».

### В объёме
- Переключатель авторов (пилюли, одно саммари за раз, состояние «одно саммари»).
- Мета автора: нейтральный аватар-инициалы, «участница клуба», время чтения
  (считается из текста), дата публикации.

### Вне объёма (сознательно отложено)
- Нумерация секций «01 …» (авто-нумерация `##`).
- Реакция «Полезно».
- Цветные аватары из пастельной палитры псевдонимов — запрещены дизайн-каноном
  (см. CLAUDE.md «Дизайн-система»). Аватары нейтральные.
- Клиентская shallow-навигация между авторами (возможная последующая оптимизация).

## Поток данных (SSR, без новых полей БД)

`app/books/[bookSlug]/summaries/page.tsx` остаётся серверным компонентом:

1. Читает `searchParams.author`.
2. Грузит `getPublishedSummariesForBook(book.id)` (как сейчас; порядок —
   `publishedAt desc, updatedAt desc`, т.е. свежие первыми).
3. Строит для каждого саммари `authorSlug` (slug из `displayName`; при коллизии
   суффикс `-2/-3…` по порядку) и `readingMinutes`.
4. Выбирает активное саммари: совпадение по `?author=<slug>`, иначе первое.
5. Рендерит: шапка книги (без изменений) → переключатель → **одно** саммари.

## Компоненты

Каждый — одна ответственность, изолирован, тестируем отдельно.

- **`lib/summary-view.ts`** — чистые хелперы без React:
  - `buildAuthorSlugs(summaries): string[]` — слаги с дедупликацией по порядку.
  - `estimateReadingMinutes(markdown): number` — слова из `bodyMarkdown`,
    `max(1, round(words / 150))`.
  - `selectSummary(summaries, slugs, param): index` — индекс по параметру;
    дефолт 0; неизвестный/пустой param → 0.
- **`components/nd/AuthorAvatar.tsx`** — нейтральный аватар-инициалы (инициалы из
  `displayName`, фон `var(--bg-tag)`, текст `var(--text)`, круг). Переиспользуется
  в пилюле и в мета-блоке.
- **`components/nd/SummaryAuthorSwitcher.tsx`** — ряд серверных
  `<Link href="?author=slug">`; активная пилюля подсвечена линией
  `var(--border-strong)` + `var(--text)`, остальные `var(--border)` +
  `var(--text-secondary)`. CTA «+ Написать своё» в конце. При одном саммари —
  строка «Пока одно саммари этой книги» + та же CTA вместо пилюль.
- **`components/nd/SummaryArticle.tsx`** — один блок саммари: мета автора
  (аватар, имя, «участница клуба», `N мин чтения`, дата) → заголовок → блок
  «В двух словах» → `SummaryMarkdown`. Выносится из текущего инлайна в `page.tsx`.

## Навигация

Клик по пилюле — обычная ссылка `?author=slug`; Next делает SSR-переход
(`export const dynamic = 'force-dynamic'` уже стоит). URL шарится, Back работает,
каждый автор индексируется.

## Граничные случаи

- `?author=` с неизвестным/пустым slug → молча первое саммари (без 404).
- Ноль опубликованных саммари → `notFound()` (как сейчас).
- Коллизия `displayName` → суффикс по порядку в `buildAuthorSlugs`.
- `canonical` в `generateMetadata` остаётся на `/summaries` без query — чтобы не
  плодить дубли в поисковом индексе; прямой `?author=` всё равно работает для
  расшаривания.

## Тестирование

- **Unit (`lib/summary-view.test.ts`):** дедуп слагов; формула времени чтения;
  выбор активного саммари (по параметру / дефолт / неизвестный).
- **E2E (`e2e/book-summaries.spec.ts`):**
  - клик по второй пилюле меняет видимое саммари;
  - `?author=` сохраняется после `page.reload()` (персистентность URL-состояния);
  - состояние «одно саммари» показывает строку-заметку, а не пилюли.

## Дизайн-канон

Острые углы (кроме круга аватара), линия вместо заливки для активного состояния,
только токены `var(--…)`, нейтральные аватары. См. CLAUDE.md «Дизайн-система».
