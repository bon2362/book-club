# Реакция «Полезно» для саммари

**Issue:** [#426](https://github.com/bon2362/book-club/issues/426)

## 1. Цель

Добавить к каждому опубликованному саммари необязательную реакцию «Полезно».
Она служит тёплым социальным сигналом: помогает читателям поддержать автора и
показывает, что текст оказался полезным. Реакция доступна без регистрации, но
по возможности считает не клики, а уникальных людей.

Точная идентификация анонимного человека невозможна. В продукте принимается
приближение:

- вошедший пользователь считается по внутреннему `user.id` и дедуплицируется
  между устройствами;
- гость считается по случайному first-party cookie и дедуплицируется в пределах
  одного браузерного профиля;
- очистка cookie или другой браузер создают новую гостевую идентичность;
- несколько людей в одном браузерном профиле могут считаться одним гостем.
- после переноса реакции в аккаунт и последующего выхода сайт снова видит
  посетителя как гостя; без авторизации невозможно доказать, что это тот же
  человек, поэтому он сможет создать новую гостевую реакцию.

Публичный UI поэтому не утверждает «N человек», а показывает только
`Полезно · N`.

## 2. Границы v1

Входит в v1:

- добавить и снять реакцию;
- показать количество активных реакций;
- разрешить реакцию гостю и вошедшему пользователю;
- объединить реакции текущего гостевого браузера с аккаунтом после входа;
- сохранять гостевое состояние после перезагрузки страницы;
- обеспечить уникальность на уровне БД;
- документировать cookie и обработку псевдонимного идентификатора.

Не входит в v1:

- CAPTCHA, browser fingerprinting и отдельный anti-bot сервис;
- IP как идентификатор или сохраняемое поле;
- рейтинг авторов, сортировка саммари по реакциям или модерационные решения;
- список пользователей, поставивших реакцию;
- уведомление автора;
- отдельные PostHog-события;
- раздельные «гостевой» и «подтверждённый» счётчики.

## 3. Пользовательский контракт

- При нуле кнопка называется `Полезно`, без `· 0`.
- При положительном значении кнопка называется `Полезно · N`.
- Неактивная кнопка добавляет реакцию, активная снимает её.
- Активное состояние имеет `aria-pressed="true"` и визуальное акцентное
  оформление из существующих токенов дизайн-системы.
- Во время запроса повторное нажатие блокируется.
- UI обновляется оптимистично, а затем принимает авторитетные `count` и
  `reacted` из ответа сервера.
- При ошибке оптимистичное изменение откатывается и рядом появляется короткое
  сообщение `Не получилось. Попробуйте ещё раз.`
- Реагировать можно только на `book_summaries.status = 'published'`. Для
  отсутствующего или неопубликованного саммари API отвечает `404`, не раскрывая
  существование черновика.

Реакция располагается в нижнем футере активного `SummaryArticle`, после текста
саммари. Переключение автора меняет и кнопку, и счётчик, потому что реакция
принадлежит конкретному `summary.id`, а не книге целиком.

## 4. Cookie гостя

### 4.1 Свойства

Cookie создаётся сервером только после первого успешного гостевого `PUT`:

- имя: `__Secure-summary-helpful`;
- значение: случайный UUID с достаточной энтропией;
- `HttpOnly`;
- `Secure`;
- `SameSite=Lax`;
- без `Domain`, то есть host-only;
- `Path=/api/summaries`;
- `Max-Age=31536000` — 12 месяцев.

Клиентский JavaScript не читает cookie. Ограниченный `Path` означает, что
браузер отправляет его только reaction API под `/api/summaries`, а не каталогу,
профилю, auth-маршрутам и остальным страницам сайта.

### 4.2 Sliding expiration

Новый cookie не создаётся от простого просмотра страницы. Если cookie уже есть,
сервер повторно устанавливает его с `Max-Age=31536000` при успешном обращении к
reaction API — чтении состояния, добавлении, снятии или объединении реакций.
Таким образом срок продлевается только при использовании страницы саммари, а не
при любой активности на сайте.

Если гость не открывал саммари больше 12 месяцев, cookie исчезает. Сама реакция
остаётся частью агрегированного счётчика, но прежний браузер больше не может
распознать и снять её. Это осознанный компромисс между долговечностью социального
сигнала и сроком жизни псевдонимного идентификатора.

### 4.3 Хранение идентификатора

Сервер проверяет формат UUID и вычисляет SHA-256. В БД сохраняется только
`visitor_hash`; исходное значение cookie не сохраняется и не логируется. Хэш
используется исключительно для реакций на саммари и не передаётся в PostHog.

## 5. Модель данных

Новая таблица `book_summary_helpful_reactions`:

| Поле | Тип | Правило |
| --- | --- | --- |
| `id` | `text` | UUID primary key |
| `summary_id` | `text` | FK → `book_summaries.id`, `ON DELETE CASCADE` |
| `user_id` | `text nullable` | FK → `user.id`, `ON DELETE CASCADE` |
| `visitor_hash` | `text nullable` | SHA-256 гостевого UUID |
| `created_at` | `timestamp` | `NOT NULL DEFAULT now()` |

Check constraint требует, чтобы была заполнена ровно одна идентичность:
`user_id` XOR `visitor_hash`.

Частичные unique-индексы:

- `unique(summary_id, user_id) where user_id is not null`;
- `unique(summary_id, visitor_hash) where visitor_hash is not null`.

Индексы делают добавление идемпотентным и закрывают гонку двух параллельных
запросов. Публичный счётчик равен `count(*)` строк конкретного саммари.

При удалении аккаунта его реакции удаляются каскадно. При удалении саммари
удаляются все связанные реакции.

## 6. Бизнес-слой

Новый модуль `lib/summary-helpful.ts` изолирует работу с идентичностью и БД:

- валидирует и хэширует гостевой UUID;
- проверяет, что саммари опубликовано;
- возвращает публичный счётчик и личное состояние;
- идемпотентно добавляет или снимает реакцию;
- объединяет гостевые реакции текущего cookie с аккаунтом;
- оборачивает все `insert`/`delete` в `withAuditContext`.

Тип идентичности бизнес-слоя — явный discriminated union:

```ts
type HelpfulActor =
  | { kind: 'user'; userId: string; visitorHash?: string }
  | { kind: 'visitor'; visitorHash: string }
  | { kind: 'new-visitor'; visitorId: string; visitorHash: string }
```

Route handlers отвечают за session/cookie/HTTP, а модуль — за правила и
транзакции. Это позволяет тестировать дедупликацию без Next.js runtime.

## 7. API

### `GET /api/summaries/{id}/helpful`

Возвращает состояние кнопки:

```json
{ "count": 12, "reacted": true }
```

- Для вошедшего пользователя `reacted` определяется по `user.id`.
- Для гостя без cookie возвращается `reacted=false`, новый cookie не создаётся.
- Для гостя с cookie используется `visitor_hash`, а cookie получает новый
  12-месячный срок.
- Ответ имеет `Cache-Control: private, no-store`, потому что `reacted`
  персонализирован.

### `PUT /api/summaries/{id}/helpful`

Идемпотентно устанавливает реакцию и возвращает актуальные `{ count, reacted }`.

- Гостю без cookie сервер генерирует UUID, добавляет реакцию по его хэшу и
  устанавливает cookie только после успешной транзакции.
- Гостю с cookie повторный `PUT` не создаёт вторую строку.
- Для вошедшего пользователя используется `user.id`; если запрос также принёс
  старый гостевой cookie, перед добавлением выполняется объединение.

### `DELETE /api/summaries/{id}/helpful`

Идемпотентно снимает реакцию и возвращает актуальные `{ count, reacted:false }`.
Для вошедшего пользователя запрос сначала объединяет текущую гостевую
идентичность, затем удаляет реакцию аккаунта, чтобы один клик действительно снял
видимую активную реакцию.

### `POST /api/summaries/helpful/reconcile`

Требует активную сессию. Принимает `summaryId`, нужный только для возвращаемого
состояния текущей кнопки. Если есть гостевой cookie:

1. находит все реакции с его `visitor_hash`;
2. для каждого саммари вставляет реакцию `user_id` с `ON CONFLICT DO NOTHING`;
3. удаляет гостевые строки;
4. возвращает `{ count, reacted }` для переданного `summaryId`;
5. удаляет гостевой cookie после успешной транзакции — аккаунту он больше не
   нужен.

Вся операция атомарна. Если аккаунт уже реагировал на то же саммари с другого
устройства, две строки схлопываются в одну. Без сессии endpoint отвечает `401`.

Все mutation endpoints используют методы `PUT`, `DELETE` или same-site `POST`,
не включают CORS и полагаются на `SameSite=Lax` cookies. Тело и ответы описываются
в `public/openapi.json`.

## 8. Серверный и клиентский поток

Публичная страница продолжает серверно получать активное саммари. Дополнительно
она запрашивает его публичный `helpfulCount` и передаёт в `SummaryArticle`:

- `summaryId`;
- `initialHelpfulCount`;
- признак активной сессии.

`SummaryArticle` рендерит клиентский `SummaryHelpfulButton`. Начальный счётчик
виден сразу и не вызывает скачка вёрстки. До загрузки личного состояния кнопка
помечена `aria-busy=true` и недоступна для нажатия.

После hydration:

- гость вызывает `GET /api/summaries/{id}/helpful`;
- вошедший пользователь вызывает `POST /api/summaries/helpful/reconcile`, который
  одновременно объединяет cookie и возвращает состояние активного саммари;
- при отсутствии или ошибке reconcile клиент может безопасно вызвать `GET`, а
  последующий `PUT`/`DELETE` всё равно повторит объединение перед мутацией.

На клик компонент оптимистично меняет `reacted` и счётчик на `±1`, блокирует
повторное нажатие, затем заменяет локальные значения ответом сервера. При ошибке
возвращает предыдущие значения.

## 9. Аудит и приватность

`book_summary_helpful_reactions` — новая мутабельная таблица, поэтому она:

- добавляется в `AUDITED_TABLES`;
- получает audit-trigger в новой миграции;
- мутируется только внутри `withAuditContext` с `source='summary-helpful'`;
- для гостя использует `actorUserId=null`, для аккаунта — текущий `user.id`.

`visitor_hash` является псевдонимным идентификатором. Та же миграция обновляет
`audit_capture()` так, чтобы перед записью `before`/`after` удалять
`visitor_hash` для этой таблицы. Хэш не должен оказаться в audit log, логах API,
ошибках или PostHog.

Privacy-документация явно сообщает:

- назначение cookie — запомнить реакцию и не считать повторный клик;
- cookie first-party, HttpOnly и ограничен reaction API;
- срок — 12 месяцев с последнего открытия саммари;
- гость может снять реакцию, пока браузер хранит cookie;
- после входа реакции переносятся к аккаунту и подчиняются удалению аккаунта.

## 10. Ошибки и конкурентность

- Два параллельных `PUT` защищены unique-индексом и возвращают одну реакцию.
- Повторный `DELETE` успешен и возвращает `reacted=false`.
- Невалидный cookie не используется и не продлевается; при следующем гостевом
  `PUT` сервер выдаёт новый UUID.
- Ошибка БД не должна устанавливать новый cookie или показывать оптимистичный
  результат как сохранённый.
- Ошибка reconcile не удаляет cookie и не оставляет частично перенесённые строки.
- Счётчик в ответе всегда читается после мутации/объединения и является
  авторитетным при конкурирующих кликах.
- Rate limiting и CAPTCHA намеренно отсутствуют: метрика не влияет на деньги,
  права, рейтинг или модерацию. При появлении реальной накрутки защита добавляется
  отдельным изменением по наблюдаемым данным.

## 11. Тестирование

### Unit и API

- UUID validation и стабильный SHA-256;
- state/count для гостя, аккаунта и гостя без cookie;
- первый гостевой `PUT` устанавливает cookie только после успеха;
- повторный `PUT` не увеличивает счётчик;
- повторный `DELETE` остаётся успешным;
- неопубликованное/несуществующее саммари возвращает `404`;
- reconcile переносит несколько гостевых реакций к аккаунту;
- конфликт с уже существующей реакцией аккаунта схлопывается;
- ошибка внутри reconcile откатывает всю транзакцию и сохраняет cookie;
- `visitor_hash` маскируется функцией аудита;
- реестр `AUDITED_TABLES` и триггеры остаются синхронны;
- `SummaryHelpfulButton` показывает нулевую и ненулевую подпись, optimistic state,
  pending и rollback при ошибке;
- `SummaryArticle` передаёт идентификатор и начальный счётчик кнопке.

### E2E

Перед написанием Playwright-теста обязательно прочитать
`docs/features/testing.md` и использовать изолированную Neon-ветку `e2e`.

Сценарий гостя:

1. создать опубликованное test-summary через фикстуру;
2. увидеть `Полезно` без `· 0`;
3. нажать и увидеть `Полезно · 1` в активном состоянии;
4. выполнить `page.reload()` и подтвердить сохранённое активное состояние;
5. снять реакцию;
6. снова выполнить `page.reload()` и подтвердить `Полезно` без счётчика;
7. cleanup удаляет test-summary и каскадно реакции.

Сценарий объединения покрывается API/integration-тестом: он требует двух
идентичностей и надёжнее проверяет атомарный conflict path, чем UI-авторизация.

Поскольку добавляется новый UI-флоу и персистентное состояние, E2E обязателен.
Положение футера и отсутствие layout shift проверяются в
`e2e/ui-states.spec.ts` через `boundingBox()`.

Реализация обновляет:

- `docs/features/book-summaries.md` — схема, API, cookie и бизнес-правила;
- `docs/wiki/Book-Summaries.md` — поведение для владельца проекта;

  связь с аккаунтом;
- `public/openapi.json` — четыре публичных HTTP-операции;
- migration/audit tests и E2E fixtures.

Отдельный feature flag не нужен. Миграция создаёт пустую таблицу, поэтому rollout
обратимо начинается со счётчика ноль. Если UI придётся временно убрать, данные
реакций можно сохранить без влияния на остальные части сайта.

## 13. Критерии готовности

- Гость может добавить и снять одну реакцию без регистрации.
- Reload сохраняет состояние гостя.
- Cookie не создаётся до первого клика и продлевается только reaction API.
- Один аккаунт учитывается один раз между устройствами.
- Вход объединяет реакции текущего браузера без увеличения счётчика.
- Неопубликованные саммари не принимают реакции.
- `visitor_hash` отсутствует в audit log и логах приложения.
- Unit, route, component, E2E и layout-тесты зелёные.
- Lint, typecheck, Jest, build и обязательный Playwright-сценарий проходят.
- OpenAPI, feature docs, Wiki и privacy-текст соответствуют реализации.
