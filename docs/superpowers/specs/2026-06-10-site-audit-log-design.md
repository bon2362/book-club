# Site-wide audit log — дизайн

**Дата:** 2026-06-10
**Цель:** системно логировать историю изменений на сайте (кто, что, где, когда, старое→новое значение) для расследования инцидентов и жалоб пользователей.
**Связанный issue:** [#344](https://github.com/bon2362/book-club/issues/344) — снапшоты расчёта сценариев (частный случай, отдельный стор, см. §9).

---

## 1. Решение по парадигме: audit log, не event sourcing

Запрос звучал как «изучить применимость event sourcing». Вывод: **event sourcing избыточен** для поставленной цели.

- **Event sourcing** делает события *единственным* источником истины, а текущее состояние — проекцией, пересобираемой реплеем. Оправдан, когда состояние обязано выводиться из событий (балансы, CRDT, саги). Цена: переписать весь слой записи, версионировать схему событий, держать проекции и снапшоты.
- **Audit log (выбранный подход):** таблицы состояния (`books`, `users`, `book_priorities`…) остаются источником истины. Рядом ведётся append-only журнал «кто/что/где/когда, before→after». Это ровно то, что нужно для расследований; ES дал бы то же самое ценой несоразмерной сложности.

Прецеденты в проекте, на которых строимся: `matchingPreferenceEvents` (уже audit-log-образная таблица с `before`/`after`/`actor`), `userActivityEvents`, `frozenScenarioJson`.

## 2. Решение по хранилищу: Postgres, не Elastic/CDC

Рассмотрены «тяжёлые» инструменты по-крупному, без оглядки на бюджет:

| Инструмент | Какую боль решает | Есть ли она у проекта |
|---|---|---|
| Debezium/Kafka (CDC) | Логировать БД, к которой обращается *много сервисов* мимо вашего кода | Нет — одно Next.js-приложение |
| Elasticsearch/Kibana | Поиск по сотням тысяч событий/день, дашборды для аналитиков | Нет — десятки изменений/день, расследует один админ |
| Temporal tables | Реконструкция полного состояния на момент T | Не требуется (цель — только расследования) |
| Datadog/Loki | Корреляция логов десятков сервисов | Нет; короткий retention и нетранзакционность против цели |

**Ключевой довод против внешних приёмников:** журнал расследований должен быть полным и достоверным. Postgres-таблица коммитится в одной транзакции с самим изменением → если изменение произошло, запись в журнале гарантированно есть. Любой внешний приёмник (Elastic, Datadog) — «доставлю позже, если ничего не упадёт» → потенциальные дыры именно там, где они недопустимы.

`audit_log` спроектирован так, чтобы при будущем росте легко стать источником для outbox→Elastic без переделки.

## 3. Решение по механизму захвата: triggers-first

Главный риск app-level аудита — разработчик (человек или AI-агент) забудет добавить запись в лог при новой фиче. Полагаться на дисциплину нельзя. Поэтому захват делает **сама БД**:

- **Триггеры Postgres на всех аудируемых таблицах** автоматически пишут `before`/`after` (`row_to_json`) в `audit_log` при любом INSERT/UPDATE/DELETE. Полнота гарантирована на уровне хранилища — ничто не проскакивает, включая ручной SQL.
- **Код сообщает только «кто/откуда/зачем»** через transaction-local настройку (`SET LOCAL`), которую триггер читает. Драйвер `drizzle-orm/neon-serverless` (Pool/WebSocket) держит сессию внутри `db.transaction()`, поэтому `SET LOCAL` доживает до триггера.

Принцип: **полнота — обязанность БД, богатый контекст — обязанность кода.** Забывчивость «ломается безопасно»: изменение всё равно записано, теряется максимум имя актора, и такие записи заметны (см. §6).

## 4. Модель данных — таблица `audit_log`

```ts
export const auditLog = pgTable('audit_log', {
  id:            text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  occurredAt:    timestamp('occurred_at', { mode: 'date' }).notNull().defaultNow(),
  actorUserId:   text('actor_user_id'), // БЕЗ FK: append-only журнал не мутируется каскадом ON DELETE; null = system/внесистемное
  actorLabel:    text('actor_label'),   // снимок «кто» (имя/email на момент действия) — переживает удаление юзера
  source:        text('source').notNull(),       // 'admin'|'api'|'cron'|'auth'|'profile'|'trigger'
  action:        text('action').notNull(),        // 'create'|'update'|'delete'
  entityType:    text('entity_type').notNull(),   // имя таблицы: 'books'|'user'|'book_priorities'|…
  entityId:      text('entity_id'),               // PK затронутой строки (композитные — сериализуем)
  before:        jsonb('before'),
  after:         jsonb('after'),
  changedFields: jsonb('changed_fields').$type<string[]>(), // удобство: какие ключи сменились
  reason:        text('reason'),                  // опц. человеческая причина (напр. текст отклонения заявки)
  metadata:      jsonb('metadata'),               // requestId, ip, userAgent, stateVersion (для матчинга)
}, (t) => ({
  entityIdx: index('audit_log_entity_idx').on(t.entityType, t.entityId, t.occurredAt), // история объекта
  actorIdx:  index('audit_log_actor_idx').on(t.actorUserId, t.occurredAt),             // что делал актор
  timeIdx:   index('audit_log_occurred_at_idx').on(t.occurredAt),                      // недавняя активность
}))
```

`actorLabel` — намеренная денормализация: при расследовании спустя месяцы юзер мог удалиться (FK `set null`), а текстовый снимок «кто» сохранится.

Хранится **полная строка** в `before`/`after` (а не только diff): проще реализуется триггером (`row_to_json`), `changedFields` вычисляется поверх. Объём на масштабе проекта незначим.

## 5. Захват: триггер + контекст из кода

### 5.1 Триггерная функция (одна, общая)
plpgsql-функция `audit_capture()`:
- на INSERT/UPDATE/DELETE пишет строку в `audit_log` с `entityType = TG_TABLE_NAME`, `action` по `TG_OP`, `before = row_to_json(OLD)`, `after = row_to_json(NEW)`;
- `actorUserId`/`actorLabel`/`source`/`reason` читает из `current_setting('app.audit_actor', true)` и т.п.; если настройка пуста → `actorUserId = NULL`, `source = 'trigger'` (внесистемная запись);
- `entityId` извлекает из PK строки (для композитных ключей — конкатенация).

Триггеры навешиваются на каждую таблицу из реестра (§7) отдельной SQL-миграцией.

### 5.2 Контекст из кода — `withAuditContext`
```ts
withAuditContext(
  { actorUserId, actorLabel, source, reason? },
  async (tx) => { /* обычные drizzle-записи через tx */ },
)
```
Открывает `db.transaction()`, делает `SET LOCAL app.audit_actor = …` (+ source/label/reason), выполняет тело. Все мутации внутри подхватываются триггером с контекстом. Один враппер на запрос — больше «на каждой мутации» помнить нечего.

### 5.3 Опциональная заметка (reason)
`reason` передаётся через тот же контекст (`SET LOCAL app.audit_reason`) там, где есть человеческая причина (например, текст отклонения заявки). Это не отдельный механизм — расширение контекста.

### 5.4 Маскирование секретов (по результату ревью)
`to_jsonb(NEW)` пишет всю строку, поэтому триггер **вырезает чувствительные ключи до записи**: `verificationToken.token` (действующий magic-link токен — иначе админ-просмотрщик показывал бы валидный вход), `telegram_preauth_tokens.token_hash`. Маска — в `audit_capture()` по `TG_TABLE_NAME` (`v_after := v_after - 'token'`). Полный охват таблиц сохраняется; теряется только секрет, не факт изменения. Список масок расширяется в одном месте (функция триггера).

### 5.5 Неизменяемость (append-only)
FK с `actor_user_id` снят намеренно: `ON DELETE set null` требовал бы UPDATE по `audit_log` и конфликтовал с неизменяемостью. Это позволяет `REVOKE UPDATE, DELETE ON audit_log` для defense-in-depth (если роль приложения не owner; иначе неизменяемость держится на ESLint-запрете мутаций `audit_log` из кода).

## 6. Защита от забывания (drift prevention)

Защита в глубину, от сильной к слабой:

1. **Полнота — на БД (триггеры).** Забыли всё в коде → изменение всё равно записано.
2. **«Кто» — централизованно (один `withAuditContext` на запрос),** а не на каждой мутации.
3. **Безопасная и громкая поломка.** Мутация вне контекста → запись с `source='trigger'`, `actor=null`. Данные не потеряны; «неизвестные» записи — сигнал. Поднимаем его:
   - CI/e2e-проверка: прогнать ключевые роуты на тест-БД и убедиться, что для нормального трафика не появляется `source='trigger'`-записей;
   - бейдж в админ-просмотрщике: «N внесистемных записей».
4. **Единый реестр + lint.** `AUDITED_TABLES` — один источник для (а) генерации триггер-миграций, (б) ESLint-правила, запрещающего сырые `db.insert/update/delete` мимо `withAuditContext` вне разрешённых модулей, (в) reconciliation-теста. Новая мутабельная таблица → одна строка в реестр → проверки и триггеры подцепляются.
5. **Процесс.** Правило в `CLAUDE.md`: «новая мутабельная таблица → добавить в `AUDITED_TABLES`». Страховка к структурным слоям, не замена.

## 7. Охват и реестр

```ts
export const AUDITED_TABLES = [
  'books', 'user', 'book_priorities', 'book_submissions',
  'intro_sections', 'signup_books', 'feedback', 'tag_descriptions',
  'matching_sessions', 'matching_session_participants',
  'matching_pseudonym_reservations', 'matching_preference_events',
  'user_activity_events', 'user_identities',
  'verificationToken', 'telegram_preauth_tokens', 'notification_queue',
  // …дополняется при росте схемы (см. §6.4 — реестр обязателен к обновлению)
] as const
```

**Полный охват — логируем все мутабельные таблицы**, включая технические/эфемерные (токены верификации, telegram-preauth, очередь нотификаций, резервации псевдонимов). Решение: ничто не исключается — для расследований лучше избыточный журнал, чем пропущенная ниточка. Эфемерные таблицы дадут больше шума, но «камера на двери» снимает их автоматически (нулевая цена в коде), а фильтрация шума — задача просмотрщика (§10), а не захвата.

> Замечание: `audit_log` сам в реестр **не входит** (иначе триггер логировал бы собственные вставки — бесконечная рекурсия). Это единственная мутабельная таблица-исключение, по техпричине.

## 8. Матчинг: зеркалирование

`matchingPreferenceEvents` остаётся доменным стором матчинга — кормит:
- **пользовательскую ленту** (вычисляемую проекцию: смена лидера-сценария, adrift; на псевдонимах);
- **админскую «Аналитику изменений предпочтений»**.

Ничего из этого не меняется. Дополнительно мутации матчинга **зеркалятся** в `audit_log` (через триггеры на `matching_*`-таблицах + контекст с реальным actor), чтобы у админа было единое окно расследований по всему сайту с реальными личностями. `stateVersion` кладётся в `metadata` для связки с §9.

**Видимость для пользователя — свойство проекции, а не строки.** Пользователь физически читает другую модель (вычисленную ленту), а не `audit_log`. Требование «до пользователей доходит только ограниченная часть» выполняется структурно: `audit_log` — admin-only по построению.

## 9. Связь с issue #344

`audit_log` отвечает на «кто/что поменял» (diff мутаций). #344 — снимок *вычисленного выхода* (сценарии на каждый `stateVersion`): другой объём и форма. Остаётся **отдельным snapshot-стором**, но связывается через `metadata.stateVersion` в `audit_log` → join «эта перестановка рангов → вот такой снимок сценариев». Не сливаем.

## 10. Просмотрщик и retention

- Новая вкладка в админке: фильтры по actor / entityType / entityId / диапазону дат; строка раскрывается в `before`→`after` diff (подсветка `changedFields`); бейдж внесистемных записей (§6.3).
- Доступ — только `isAdmin` (как остальные admin API).
- Retention: на старте **без удаления** (объём копеечный). Cron-архивацию не делаем (YAGNI); заложить как будущий путь.

## 11. Тестирование

- **Unit:** `withAuditContext` (формирование `SET LOCAL`), хелпер `diffRow`/извлечение `changedFields` (edge cases: create без before, частичный update, композитный ключ).
- **E2E (изолированная ветка `e2e`):** админ меняет видимость книги → запись появляется в просмотрщике с верными `before/after` и actor; **после `page.reload()`** запись на месте (персистентность). Отдельный тест: мутация вне `withAuditContext` → запись `source='trigger'` с `actor=null` (проверка страховочной сети).
- **Reconciliation-тест (CI):** ключевые роуты не порождают внесистемных записей для нормального трафика.

## 13. Решения, принятые на этапе плана (ранее открытые)

- **Охват:** полный, ничего не исключаем; секреты маскируются в триггере (§5.4).
- **`entityId` для композитных PK:** триггер собирает составной ключ `session_id:user_id:book_id` из доступных колонок (иначе индекс по `entity_id` был бы бесполезен для самых мутируемых `book_priorities`/`signup_books`).
- **Где ставить `withAuditContext`:** на каждом мутирующем роуте (actor из сессии запроса), не глобальной обёрткой.
- **Auth-пути (NextAuth DrizzleAdapter):** пишут мимо нашего кода → останутся `source='trigger'`. Auth-таблицы входят в allowlist «легитимных внесистемных» (`AUTH_OOB_TABLES`), чтобы reconciliation-сигнал не тонул в логинах.
- **Транзакционная семантика `set_config`:** доказывается spike-проверкой на живой БД **до** обвязки роутов (несущее допущение всего механизма).
