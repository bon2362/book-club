# Feature Documentation + GitHub Wiki Auto-Sync — Design Spec

**Issue:** #76
**Date:** 2026-03-21
**Status:** Approved

---

## Overview

Create `docs/features/` with concise markdown files describing the current implementation of each major feature area, and a GitHub Actions workflow that automatically syncs these files to the GitHub Wiki on every push to `main`.

**Audience:** Claude Code sessions (quick context loading), the developer (reference), future contributors (onboarding).

---

## 1. docs/features/ Structure

Five files, one per feature area. Each file is ~1 page (concise, current state only — not history).

### Files

| File | Covers |
|------|--------|
| `auth.md` | Google OAuth flow, magic link, NextAuth v5 config, JWT callbacks, `isAdmin` flag, protected routes |
| `books-catalog.md` | Google Sheets as data source, `coverUrl` from column L, `CoverImage` fallback, `BookCard` expand/collapse |
| `admin-panel.md` | `isAdmin` check, member/book management, status widgets, book priorities display |
| `notifications.md` | `notification_queue` table, GitHub Actions cron (`digest.yml`), Resend email, status model |
| `user-profile.md` | Drawer component, tabs (profile/priorities/languages), `ContactsForm`, data persistence |

### File Format

Each file follows this structure:

```markdown
# Feature Name

## What it does
Short description of user-facing behaviour.

## How it works
Data flow and key architectural decisions. 2-5 bullet points.

## Key files
- `path/to/file.ts` — what it does
- `path/to/component.tsx` — what it does
```

## 2. GitHub Actions Workflow

**File:** `.github/workflows/wiki-sync.yml`

### Trigger

```yaml
on:
  push:
    branches: [main]
    paths:
      - 'docs/features/**'
```

Only fires when `docs/features/` files change — no unnecessary runs.

### Steps

1. **Checkout** the repo
2. **Enable Wiki** via GitHub API — `PATCH /repos/bon2362/book-club` with `{"has_wiki": true}` — idempotent, safe to run every time
3. **Clone wiki repo** using `GITHUB_TOKEN` — `https://x-access-token:${{ secrets.GITHUB_TOKEN }}@github.com/bon2362/book-club.wiki.git`
4. **Copy** `docs/features/*.md` into the cloned wiki directory (overwrites everything)
5. **Generate `Home.md`** — auto-generated table of contents with links and last-updated date
6. **Commit and push** to wiki repo

### Authentication

Uses the built-in `secrets.GITHUB_TOKEN` — it has write access to the wiki of the same repository. No additional secrets needed.

### Overwrite Strategy

Wiki is always fully overwritten from `docs/features/`. Manual edits to the wiki are lost on next push. This is intentional — `docs/features/` is the single source of truth.

## 3. Home.md Generation

Generated by the workflow script inline (bash heredoc or simple script). Structure:

```markdown
# Book Club — Feature Documentation

> Auto-generated from `docs/features/`. Last updated: YYYY-MM-DD.
> To edit, modify files in `docs/features/` and push to `main`.

## Features

- [Authentication](auth) — Google OAuth, magic link, NextAuth v5
- [Books Catalog](books-catalog) — Google Sheets, covers, BookCard
- [Admin Panel](admin-panel) — moderation, priorities, stats
- [Notifications](notifications) — digest queue, cron, Resend
- [User Profile](user-profile) — drawer, tabs, priorities
```

Links use filenames without `.md` extension (GitHub Wiki requirement).

## 4. Enabling Wiki

The workflow attempts to enable Wiki via `PATCH /repos/bon2362/book-club` with `has_wiki: true`. However, the default `GITHUB_TOKEN` in GitHub Actions typically lacks `administration` scope, so this step may silently fail.

**Practical approach:** Enable the Wiki manually once in GitHub repository Settings → General → Features → Wikis. After that, the workflow clone/push steps will work with standard `GITHUB_TOKEN` permissions. The PATCH step can remain in the workflow as a no-op idempotent call, or be removed.

---

## Implementation Order

1. Write the five `docs/features/*.md` files (read existing code to populate accurately)
2. Create `.github/workflows/wiki-sync.yml`
3. Push to `main` — workflow runs, Wiki gets populated automatically

---

## Out of Scope

- Per-page versioning or change history in Wiki
- Cross-links between feature pages
- Automated content extraction from code (JSDoc, etc.)
- Wiki pages for non-feature topics (setup, deployment, contributing)
