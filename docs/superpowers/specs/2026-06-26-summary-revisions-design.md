# Редактирование опубликованных саммари — дизайн

**Дата:** 2026-06-26  
**Цель:** позволить автору предложить изменения в опубликованное саммари, не скрывая текущую публичную версию до повторной модерации.  
**Связанный issue:** [#409](https://github.com/bon2362/book-club/issues/409), follow-up к #16.

## 1. Проблема

Страница `/summaries/{id}/edit` существует и для опубликованного саммари, но поля заблокированы, сервер отклоняет `PATCH`, а статус в шапке ошибочно подписан как `Черновик`. Такое поведение соответствовало исходным границам MVP, однако выглядит как поломка и не даёт автору исправить уже опубликованный текст.

Прямое редактирование строки `book_summaries` автором неприемлемо: изменение немедленно попадёт на публичную страницу и обойдёт администратора.

## 2. Продуктовое решение

Опубликованная версия остаётся в `book_summaries` и продолжает отображаться публично. Предлагаемые автором изменения хранятся в отдельной активной ревизии и проходят повторную модерацию.

Путь автора:

1. Автор открывает опубликованное саммари в редакторе.
2. Нажимает `Редактировать`.
3. Система создаёт активную ревизию, предзаполненную опубликованными полями.
4. Автор редактирует её с теми же Markdown toolbar, preview и автосейвом.
5. После отправки ревизия переходит в `pending`.
6. Пока ревизия проверяется, публичная страница продолжает показывать прежний текст.
7. При approve новая версия атомарно заменяет поля опубликованного саммари.
8. При reject публичный текст не меняется; автор видит комментарий и дорабатывает ту же ревизию.

## 3. Инварианты

- У пользователя по-прежнему не более одного `book_summaries` на книгу.
- У опубликованного саммари не более одной активной ревизии.
- Ревизия не является вторым публичным саммари и не участвует в `summaryCount`.
- Создать или отправить ревизию может только автор саммари, у которого книга всё ещё отмечена как `personal_status='read'`.
- Автор редактирует ревизию только в статусах `draft` и `rejected`.
- `pending` доступен автору только для чтения.
- Публичные выборки читают только `book_summaries`; незавершённые ревизии никогда не попадают на публичную страницу.
- Администратор сохраняет право напрямую исправлять опубликованный текст.

## 4. Модель данных

Добавляется таблица `book_summary_revisions`:

```ts
export const bookSummaryRevisions = pgTable('book_summary_revisions', {
  id: text('id').primaryKey().$defaultFn(() => crypto.randomUUID()),
  summaryId: text('summary_id').notNull()
    .references(() => bookSummaries.id, { onDelete: 'cascade' }),
  displayName: text('display_name').notNull(),
  title: text('title').notNull(),
  tldr: text('tldr').notNull(),
  bodyMarkdown: text('body_markdown').notNull(),
  status: text('status').notNull().default('draft'),
  rejectionReason: text('rejection_reason'),
  submittedAt: timestamp('submitted_at', { mode: 'date' }),
  createdAt: timestamp('created_at', { mode: 'date' }).notNull().defaultNow(),
  updatedAt: timestamp('updated_at', { mode: 'date' }).notNull().defaultNow(),
}, (t) => ({
  summaryUnique: uniqueIndex('book_summary_revisions_summary_unique').on(t.summaryId),
  statusIdx: index('book_summary_revisions_status_idx').on(t.status),
}))
```

Статусы ревизии: `draft | pending | rejected`.

После approve в одной audit-транзакции:

1. поля `display_name`, `title`, `tldr`, `body_markdown` копируются в `book_summaries`;
2. `book_summaries.updated_at` обновляется;
3. `book_summaries.published_at` сохраняется как дата первой публикации;
4. строка ревизии удаляется.

Удаление не теряет историю: обе мутации фиксируются audit trigger с before/after snapshot. Полноценный пользовательский архив версий не входит в scope.

Таблица добавляется в `AUDITED_TABLES`, а миграция создаёт `audit_book_summary_revisions`.

## 5. Серверная модель

`lib/book-summaries.ts` остаётся владельцем бизнес-правил и получает операции:

- получить активную ревизию по summary;
- создать или вернуть существующую ревизию;
- сохранить авторскую ревизию;
- отправить ревизию на проверку;
- получить ревизии для админки с данными книги и автора;
- сохранить админские правки ревизии;
- отклонить ревизию;
- применить ревизию к опубликованному саммари.

Создание ревизии разрешено только для `book_summaries.status='published'`. Повторный запрос возвращает существующую строку. Unique index защищает от параллельного создания дублей.

Approve проверяет полноту ревизии и выполняет update + delete внутри одного `withAuditContext`. Если транзакция не завершилась, публичный текст остаётся прежним, а ревизия сохраняется.

## 6. API

Авторские endpoints:

- `POST /api/summaries/{summaryId}/revision`  
  Создать или вернуть активную ревизию опубликованного саммари.

- `PATCH /api/summary-revisions/{revisionId}`  
  Автосохранение `draft` или `rejected`.

- `POST /api/summary-revisions/{revisionId}/submit`  
  Проверить обязательные поля и `personal_status='read'`, перевести в `pending`.

Существующие endpoints для первого draft сохраняются без изменений.

Админские endpoints:

- `GET /api/admin/summaries` возвращает опубликованные саммари и активные ревизии как единый список модерации с полями `kind: 'summary' | 'revision'` и `summaryId`.
- `PATCH /api/admin/summary-revisions/{id}` сохраняет админские правки.
- `POST /api/admin/summary-revisions/{id}/publish` применяет ревизию.
- `POST /api/admin/summary-revisions/{id}/reject` отклоняет ревизию с обязательной причиной.

OpenAPI обновляется для новых DTO и endpoints.

## 7. Авторский UI

`/summaries/{summaryId}/edit` получает опубликованное саммари и активную ревизию, если она есть.

Состояния:

- первоначальный `draft` — текущий редактируемый flow;
- первоначальный `pending` — read-only, статус `На проверке`;
- первоначальный `rejected` — редактируемый flow с комментарием;
- `published` без ревизии — поля показывают текущую публикацию read-only, CTA `Редактировать`;
- `published` + revision `draft` — редактируемая ревизия, статус `Правки: черновик`;
- `published` + revision `pending` — read-only, статус `Правки на проверке`, публичная версия остаётся доступна по ссылке;
- `published` + revision `rejected` — редактируемая ревизия с комментарием, статус `Правки отклонены`.

После нажатия `Редактировать` клиент создаёт ревизию и переключается в редактируемое состояние без смены URL. Автосейв и submit выбирают endpoint по типу редактируемой сущности.

В модалке книги для опубликованного саммари показываются два действия:

- основное `Читать саммари` ведёт на публичную страницу;
- вторичное `Редактировать` ведёт в редактор.

Меню прочитанной книги продолжает открывать `/summaries/{id}/edit`; редактор сам показывает корректное состояние.

## 8. Админский UI

Активная ревизия отображается во вкладке `Саммари` отдельной строкой с пометкой `Правки к опубликованному`.

Для ревизии администратор видит:

- текущую опубликованную версию как read-only reference;
- предложенные поля в существующем inline-редакторе;
- действия `Сохранить`, `Опубликовать правки`, `Отклонить правки`.

Фильтр `На проверке` включает pending первоначальные саммари и pending ревизии. После approve строка ревизии исчезает, а опубликованная строка обновляется. После reject строка остаётся в фильтре `Отклонённые`.

## 9. Ошибки и конкуренция

- Неавтор получает тот же нейтральный `summary not found`, без раскрытия существования объекта.
- Ревизия для неопубликованного саммари отклоняется.
- Создание/submit без статуса книги `read` отклоняется.
- PATCH для `pending` отклоняется.
- Approve или reject отсутствующей ревизии возвращает 400 по текущему API-паттерну.
- Повторный approve после успешного применения не меняет публикацию: ревизии уже нет.
- Если администратор напрямую изменил публикацию после создания ревизии, ревизия всё равно применяется целиком. В MVP конфликтное слияние и diff UI не добавляются; администратор видит обе версии перед approve.

## 10. Тестирование

Unit:

- создание ревизии копирует опубликованные поля;
- повторное создание возвращает существующую ревизию;
- авторские права и статусы проверяются;
- submit требует полные поля и `read`;
- публичные выборки игнорируют ревизии;
- approve атомарно обновляет summary и удаляет revision;
- reject сохраняет публичную версию;
- audit registry и migration остаются синхронными.

Component/API:

- редактор показывает корректные статусы вместо ложного `Черновик`;
- published state создаёт ревизию по CTA;
- autosave и submit ревизии используют новые endpoints;
- модалка показывает чтение и редактирование;
- админка различает исходное саммари и ревизию;
- route guards и validation errors покрыты тестами.

E2E:

1. Участник создаёт и публикует исходное саммари через админа.
2. Участник создаёт ревизию, редактирует её, дожидается автосейва и перезагружает страницу.
3. После submit публичная страница всё ещё показывает старый текст.
4. Администратор видит `Правки к опубликованному` и применяет их.
5. После reload публичная страница показывает новую версию.
6. Отдельный rejection path проверяет комментарий и сохранение старой публичной версии.

Изменение затрагивает UI-flow и персистентное состояние, поэтому E2E обязателен. Нового CSS-позиционирования или анимации нет; отдельный layout test в `e2e/ui-states.spec.ts` не требуется.

Миграция должна быть применена к production DB до использования нового flow. До миграции существующая публикация и чтение саммари продолжают работать; UI создания ревизии получит серверную ошибку. Деплой приложения и миграцию следует координировать, чтобы не оставлять пользователю неработающий CTA.

## 12. Вне scope

- история версий в пользовательском UI;
- сравнение изменений по словам или строкам;
- одновременные параллельные ревизии;
- комментарии и обсуждение правок;
- email-уведомления;
- самостоятельный rollback на старую версию;
- изменение правила «одно саммари пользователя на книгу».
