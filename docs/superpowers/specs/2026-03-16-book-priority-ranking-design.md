# Ранжированное голосование за книги (#41)

**Дата:** 2026-03-16
**Статус:** Approved

## Проблема

Пользователи записываются на несколько книг (в среднем 7–12), но у каждого есть более и менее желанные варианты. Когда на книгу набирается достаточно людей для группы, организатор (Евгений) должен понимать, кому именно написать первым — кто больше всего хочет читать именно эту книгу.

## Решение

Пользователь расставляет свои книги по приоритету (drag-and-drop) в личном кабинете. Администратор видит эти приоритеты в таблице участников — для каждой книги отсортированный список, кто хочет её сильнее всего.

---

## Пользовательский интерфейс

### Личный кабинет — вкладка «Записал:ась»

**Что меняется:** список книг становится drag-and-drop сортируемым. Позиция в списке = приоритет.

**Визуальная иерархия:**
- Каждая книга: кружок с номером ранга (слева) + иконка grip ⠿ + название + автор
- Топ-3: оранжевые кружки (#1 — тёмно-оранжевый, #2 — средний, #3 — светлый)
- Остальные: серые кружки
- Автосохранение при каждом изменении порядка, надпись «✓ Сохранено» справа внизу

**Подсказка-баннер:**
- Оранжевый баннер вверху списка: «Расставь книги по интересу: перетащи их так, чтобы сверху оказались те, которые хочется прочитать сильнее всего. Это поможет подобрать тебе подходящую группу.»
- Исчезает после первого drag (флаг `priorities_set` в БД или по факту изменения порядка от дефолтного)
- Отображается всем, кто ещё ни разу не сортировал

**Библиотека:** `@dnd-kit/core` + `@dnd-kit/sortable` — поддерживает touch-события для мобильных.

**Начальный порядок (до первой сортировки):** порядок записи на книги (хронологический). Баннер при этом отображается.

## Хранение данных

**Новая таблица `book_priorities`:**

```sql
CREATE TABLE book_priorities (
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  book_name  TEXT NOT NULL,
  rank       INTEGER NOT NULL,       -- 1 = наиболее желанная
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, book_name)
);
```

**Флаг «расставлял ли пользователь приоритеты»:** определяется наличием хотя бы одной записи в `book_priorities` для данного пользователя. Если записей нет — «не расставлял».

**API:**
- `GET /api/priorities` — возвращает приоритеты текущего пользователя: `[{ bookName, rank }]`
- `PUT /api/priorities` — принимает полный отсортированный список книг: `[{ bookName }]`, пересохраняет ранги (rank = index + 1)

## Администраторский интерфейс

### Вкладка «Участники» — колонка «Приоритет»

**Что добавляется:**
- Новая колонка «Приоритет» в таблицу участников
- Фильтр по книге (выпадающий список) в шапке вкладки — при выборе книги таблица показывает только записавшихся на неё, отсортированных по их приоритету для этой книги
- Без фильтра — колонка «Приоритет» скрыта или показывает прочерк (приоритет всегда относителен конкретной книги)

**Формат колонки «Приоритет»:**
- `№ X из N` — место данной книги среди всех записей пользователя (N = общее число его книг)
- Цвет: оранжевый для №1 и №2, светло-оранжевый для №3, серый для остальных
- Прочерк `—` если пользователь не расставлял приоритеты

**Разделение в таблице при фильтре по книге:**
1. Сначала — участники, расставившие приоритеты (отсортированы по рангу этой книги, от меньшего к большему)
2. Затем — серая строка-разделитель: «Не расставили приоритеты (N участников)»
3. Далее — участники без приоритетов (в произвольном порядке, серым цветом)

---

## Граничные случаи

| Ситуация | Поведение |
|---|---|
| Пользователь ещё не расставлял приоритеты | Баннер-подсказка отображается; в admin — группа «Не расставили» |
| Пользователь отписался от книги | Её ранг удаляется из `book_priorities`; ранги оставшихся пересчитываются при следующем PUT |
| Пользователь добавил новую книгу | Новая книга появляется в конце списка без ранга (добавляется в `book_priorities` с rank = max+1) |
| Пользователь удалил аккаунт | Каскадное удаление из `book_priorities` через FK |

---

## Что не входит в скоуп

- Уведомления или email-напоминания расставить приоритеты (достаточно баннера в кабинете)
- Автоматическое распределение по группам на основе приоритетов (это отдельная фича #42)
- Публичный показ приоритетов другим участникам
