# Ранжированное голосование за книги (#41)

**Дата:** 2026-03-16
**Статус:** Approved

## Проблема

Пользователи записываются на несколько книг (в среднем 7–12), но у каждого есть более и менее желанные варианты. Когда на книгу набирается достаточно людей для группы, организатор (Евгений) должен понимать, кому именно написать первым — кто больше всего хочет читать именно эту книгу.

## Решение

Пользователь расставляет свои книги по приоритету (drag-and-drop) в личном кабинете. Администратор видит эти приоритеты в таблице участников — для каждой книги отсортированный список, кто хочет её сильнее всего.

---

## Пользовательский интерфейс

### Личный кабинет — вкладка «Записал:ась»

**Что меняется:** список книг становится drag-and-drop сортируемым. Позиция в списке = приоритет.

**Визуальная иерархия:**
- Каждая книга: кружок с номером ранга (слева) + иконка grip ⠿ + название + автор
- Топ-3: оранжевые кружки (#1 — тёмно-оранжевый, #2 — средний, #3 — светлый)
- Остальные: серые кружки
- Автосохранение при каждом изменении порядка — с debounce 500 мс, надпись «✓ Сохранено» после успешного PUT

**Начальный порядок (до первой сортировки):** порядок элементов в массиве `selectedBooks` из Google Sheets. Клиент всегда получает порядок так:
1. Запросить `GET /api/priorities` — вернёт `[{ bookName, rank }]` или пустой массив если ни одного нет.
2. Если массив пуст — отобразить книги в порядке `selectedBooks` из Sheets.
3. Если массив не пуст — отобразить книги в порядке рангов. Книги, которых нет в ответе (новые, добавленные после первой сортировки) — добавить в конец в порядке `selectedBooks`.

**Подсказка-баннер:**
- Оранжевый баннер вверху списка: «Расставь книги по интересу: перетащи их так, чтобы сверху оказались те, которые хочется прочитать сильнее всего. Это поможет подобрать тебе подходящую группу.»
- Баннер отображается пока `user.prioritiesSet = false` в БД.
- Скрывается после первого успешного PUT `/api/priorities` (сервер устанавливает флаг, клиент обновляет локальное состояние).

**Библиотека:** `@dnd-kit/core` + `@dnd-kit/sortable`.
- Сенсоры: `PointerSensor` (десктоп) + `TouchSensor` с `activationConstraint: { delay: 250, tolerance: 5 }` (мобильный — активация по долгому нажатию, чтобы не конфликтовать со скроллом страницы).

## Хранение данных

### Изменение в таблице `users`

Добавить колонку:

```sql
ALTER TABLE "user" ADD COLUMN priorities_set BOOLEAN NOT NULL DEFAULT false;
```

В Drizzle-схеме (`lib/db/schema.ts`):

```ts
prioritiesSet: boolean('priorities_set').notNull().default(false),
```

**Смысл флага:** `false` = пользователь ни разу не вызывал PUT `/api/priorities`. Используется исключительно для управления показом баннера-подсказки в личном кабинете.

### Новая таблица `book_priorities`

```sql
CREATE TABLE book_priorities (
  user_id    TEXT NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
  book_name  TEXT NOT NULL,
  rank       INTEGER NOT NULL,  -- 1 = наиболее желанная
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, book_name)
);
```

В Drizzle-схеме:

```ts
export const bookPriorities = pgTable('book_priorities', {
  userId:    text('user_id').notNull().references(() => users.id, { onDelete: 'cascade' }),
  bookName:  text('book_name').notNull(),
  rank:      integer('rank').notNull(),
  updatedAt: timestamp('updated_at', { mode: 'date' }).notNull().defaultNow(),
}, (t) => ({
  pk: primaryKey({ columns: [t.userId, t.bookName] }),
}))
```

`updatedAt` передаётся явно в теле каждого upsert: `{ ..., updatedAt: new Date() }`. Не полагаться на `$onUpdateFn` — он не срабатывает при `onConflictDoUpdate` в Drizzle.

**Важно:** записи вставляются только через PUT `/api/priorities`, инициированный действием пользователя. Автоматической вставки при записи на книгу нет.

### API

**`GET /api/priorities`**
- Требует авторизации: 401 если нет сессии.
- Возвращает `200` с `[{ bookName: string, rank: number }]` — только книги с сохранёнными рангами, отсортированные по `rank ASC`.
- Если у пользователя нет ни одной записи — возвращает `[]`.

**`PUT /api/priorities`**
- Требует авторизации: 401 если нет сессии.
- Тело: `{ books: string[] }` — полный упорядоченный список названий книг (от наиболее желанной к наименее). Порядок элементов = приоритет, rank = index + 1.
- Валидация: `books` должен быть массивом строк, не пустым. Если невалидно — `400`.
- Логика: upsert всех записей (`INSERT ... ON CONFLICT DO UPDATE`), удалить записи для книг, которых больше нет в списке (пользователь отписался). Установить `priorities_set = true` для пользователя.
- Возвращает `200` с `{ ok: true }`.
- Дополнительная валидация (рекомендуется, не блокирующая): проверить, что все переданные названия совпадают с `selectedBooks` пользователя из Sheets. Несовпадающие — игнорировать или вернуть `400`.
- **Debounce:** клиент вызывает PUT не чаще раза в 500 мс после последнего drag-события. Если пришли два конкурирующих запроса — сервер принимает оба (последний wins на уровне upsert), данные не повреждаются.

## Администраторский интерфейс

### Вкладка «Участники» — добавления

**Фильтр по книге:** выпадающий список книг в шапке вкладки. При выборе:
- Таблица показывает только участников, записавшихся на эту книгу.
- Участники разбиты на две группы (см. ниже).
- При отсутствии фильтра — колонка «Приоритет» скрыта (приоритет всегда относителен конкретной книги).

**Новая колонка «Приоритет»** (видна только при активном фильтре по книге):
- Формат: `№ X из N` — место данной книги среди всех N книг пользователя.
- Цвет бейджа: оранжевый для №1–2, светло-оранжевый для №3, серый для остальных.
- `—` если пользователь не расставлял приоритеты (`priorities_set = false`).

**Разбивка таблицы при фильтре по книге:**

Группа 1 — участники с приоритетами:
- Только те, у кого `priorities_set = true` и есть ранг для выбранной книги.
- Отсортированы по рангу ASC. При одинаковых рангах — вторичная сортировка по `book_priorities.updated_at` ASC (кто раньше расставил).

Группа 2 — без приоритетов:
- Строка-разделитель: «Не расставили приоритеты (N участников)»
- Участники с `priorities_set = false` — серым цветом, прочерк в колонке.

**API для admin:** существующий эндпоинт получения участников расширяется query-параметром `?book=<bookName>`. При его наличии:
- Возвращаются только участники, у которых `selectedBooks` содержит данную книгу.
- Каждый участник получает дополнительные поля:
  ```ts
  priority: number | null      // rank данной книги у пользователя; null = не расставлял
  totalBooks: number | null    // общее число книг пользователя; null = не расставлял
  prioritiesSet: boolean       // скопирован из users.priorities_set
  ```
- Фронтенд формирует строку `№ priority из totalBooks` самостоятельно.

**Источник `selectedBooks` на клиенте:** `ProfileDrawer` уже получает `selectedBooks` и `books` как пропсы от родительского компонента — дополнительных запросов не нужно. Логика слияния рангов с `selectedBooks` выполняется внутри компонента.

## Граничные случаи

| Ситуация | Поведение |
|---|---|
| Пользователь не расставлял приоритеты | `priorities_set = false`, записей в `book_priorities` нет; баннер показывается; в admin — группа «Не расставили» |
| Пользователь отписался от книги | Эндпоинт отписки (`/api/signup` с удалением книги) на сервере делает `DELETE FROM book_priorities WHERE user_id = ? AND book_name = ?`. Ранги оставшихся книг не пересчитываются автоматически — клиент при следующей загрузке drawer'а вызовет GET /api/priorities, получит актуальные ранги и пересчитает отображаемые позиции. |
| Пользователь записался на новую книгу после первой сортировки | Новая книга появляется в конце drag-списка на клиенте (слитый список: сохранённые ранги + новые книги в конце); следующий PUT сохранит её ранг |
| Пользователь удалил аккаунт | Каскадное удаление из `book_priorities` через FK; `priorities_set` удаляется вместе с записью в `users` |
| `selectedBooks` пользователя пуст | Вкладка «Записал:ась» показывает пустое состояние; приоритизировать нечего, баннер не отображается |

## Что не входит в скоуп

- Уведомления или email-напоминания расставить приоритеты (достаточно баннера в кабинете)
- Автоматическое распределение по группам (#42)
- Публичный показ приоритетов другим участникам
