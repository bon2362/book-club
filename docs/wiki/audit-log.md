# Журнал изменений (audit log)

На сайте ведётся автоматический журнал всех изменений данных. Это инструмент расследований — он позволяет ответить на вопросы «кто, что и когда изменил» при инцидентах или жалобах пользователей.

## Зачем нужен журнал

Без журнала расследовать инциденты сложно: единственная информация — текущее состояние БД. Журнал фиксирует каждое изменение, пока оно происходит, и сохраняет его навсегда. Это позволяет:

- восстановить хронологию событий («когда появилась эта запись?», «кто удалил пользователя?»);
- проверить жалобу участника («мои приоритеты не сохранились»);
- убедиться, что изменение, сделанное через админку, действительно применилось;
- отследить системные операции (cron, auth) — даже те, которые происходят без участия человека.

## Что видно в журнале

Каждая строка журнала содержит:

| Поле | Что это |
| --- | --- |
| Время | Когда произошло изменение |
| Кто | Имя или email пользователя, который совершил действие |
| Источник | Откуда пришло изменение (`admin`, `api`, `cron`, `auth`, `profile`) |
| Действие | `insert`, `update`, или `delete` |
| Объект | Какая таблица затронута (`books`, `user`, `book_priorities`, …) |
| ID объекта | Первичный ключ изменённой строки |
| До / После | Полный снимок строки до и после изменения |
| Изменённые поля | Список только тех полей, которые действительно поменялись |
| Причина | Опциональный текст (например, причина отклонения заявки) |

Журнал охватывает все мутабельные таблицы сайта: книги, пользователи, приоритеты, заявки, записи на книги, матчинг, токены, очередь уведомлений и другие.

## Как расследовать

Просмотрщик открывается в admin-панели на вкладке **«История изменений»**.

Доступные фильтры:

- **По актору** — показать всё, что сделал конкретный пользователь.
- **По типу объекта** — например, посмотреть только изменения в `books`.
- **По ID объекта** — история конкретной строки (книги, пользователя, записи).
- **По источнику** — `admin`, `api`, `cron`, `auth`, `trigger`.
- **По диапазону дат** — сузить период расследования.

Клик по строке раскрывает полный снимок «до → после» с выделением изменённых полей.

## Что означает пометка «внесистемное»

Строки с меткой **«внесистемное»** (технически `source='trigger'`) появляются, когда изменение прошло через БД, но без идентификации актора в коде сайта.

Это нормально для операций авторизации: когда пользователь входит в систему, NextAuth напрямую записывает токены верификации и Telegram-данные — эти записи выглядят как «внесистемные», и так и должно быть.

В остальных случаях «внесистемное» — сигнал для разработчика: какой-то роут не передаёт контекст «кто делает запрос».

## Маскирование секретов

Журнал хранит полные строки, но чувствительные поля вырезаются **до записи** прямо в БД:

- magic-link токен из таблицы `verificationToken` — иначе действующую ссылку для входа было бы видно в просмотрщике;
- хеш токена Telegram preauth.

Факт изменения фиксируется; сам секрет — нет.

## Кто видит журнал

Журнал доступен **только администратору**. Пользователи сайта его не видят. У пользователей есть собственная лента событий матчинга (на псевдонимах) — это другая подсистема.

## Retention (хранение)

На текущем этапе записи не удаляются. Объём изменений на сайте небольшой, место в БД копеечное. В будущем можно настроить архивацию старых записей.

## Связь с issue #344 (снимки сценариев матчинга)

Журнал фиксирует факты мутаций: «кто и что поменял в данных». Issue #344 — отдельная подсистема снимков: она сохраняет вычисленный результат (какие сценарии матчинга получились) на каждый момент изменения предпочтений. Обе системы связаны через поле `stateVersion`: по нему можно найти, какой снимок сценариев соответствовал конкретному изменению приоритетов в журнале.
