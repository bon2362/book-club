## Что хорошо сработало

1. **Email Resend fail-safe (Story 2.2).** Сбой Resend не блокирует ответ API — `try/catch` с `console.error`. Требование NFR8 выполнено корректно.

2. **Нейтральный тон письма об отклонении (Story 2.1).** «не была одобрена» вместо «отклонена» — внимательный UX-выбор, снижает негативный эффект для пользователя.

3. **leftJoin для email автора (Story 2.2 GET).** Drizzle `leftJoin(users, eq(bookSubmissions.userId, users.id))` — получаем `userEmail` одним запросом без N+1.

4. **`userEmail` merge после PATCH (Story 2.3).** `setSubmissions(prev => prev.map(s => s.id === id ? { ...d.data, userEmail: s.userEmail } : s))` — правильное решение для сохранения JOIN-поля, которое PATCH не возвращает.

5. **Тест Resend в изоляции (Story 2.2).** `jest.mock('resend', ...)` с отдельным `mockSend` — тест проверяет именно факт вызова и payload, не сам Resend.

6. **9 тестов AdminPanel (Story 2.3).** Tab, загрузка, фильтр, детальный вид, approve, reject, edit+save, loading state — полное покрытие сценариев.

## Что можно улучшить

1. **Unchecked subtasks в story-файле 2.2.** Верхний уровень задач `[x]`, подзадачи остались `[ ]`. Completion Notes подтверждают реализацию, но несоответствие сбивает с толку при чтении.

2. **Нет валидации значения `status` в PATCH.** Эндпоинт принимает любую строку как `status` без проверки допустимых значений (`pending`/`approved`/`rejected`). Маршрут за `isAdmin` guard — риск низкий, но потенциально стоит добавить.

3. **Мёртвый код в тесте GET.** `makeRequest()` определена в `app/api/admin/submissions/route.test.ts` но не используется (GET не принимает аргументов).
