## Что хорошо сработало

1. **Минимальный footprint изменений.** Вся история — расширение одного файла (`app/api/books/route.ts`). Никаких новых файлов, никаких новых зависимостей.

2. **`Promise.all` с тремя запросами.** Sheets + bookStatuses + approvedSubmissions — параллельно, с индивидуальными `.catch(() => [])` guard'ами. Сбой одного источника не ломает остальные.

3. **`makeChainable()` helper для тестов.** Решение для мока, который одновременно thenable и имеет `.where()` — переиспользуемый паттерн для Drizzle chain-моков.

4. **Решение проблемы jest.mock hoisting.** Переход от `const mockSelect = jest.fn()` в фабрике к `let mockSelect: jest.Mock` + `jest.requireMock()` в `beforeEach` — задокументированное решение, применимое в других тестах.

5. **Маппинг в формат BookWithCover.** Все поля маппятся корректно, `status: null` (не статус заявки, а reading-статус), ID не конфликтуют (числовые vs UUID).

6. **6 тестов добавлено** на новую функциональность: approved в списке, маппинг полей, topic=null→tags=[], нет approved→только Sheets, DB error guard.

## Ключевые инсайты

1. **`makeChainable()` паттерн** — стоит задокументировать как стандартный подход для Drizzle моков в тестах. Применим везде, где цепочка `select().from().where()`.

2. **jest.mock hoisting** — потенциальная ловушка во всех тестах с динамическими моками. Решение через `jest.requireMock()` в `beforeEach` надёжнее, чем инлайн в фабрике.

3. **Минималистичная история** (1 файл изменён, 1 файл тестов) — правильный размер для одной AC-задачи. Разбивка была верной.
