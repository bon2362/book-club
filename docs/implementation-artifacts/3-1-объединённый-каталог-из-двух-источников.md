# Story 3.1: Объединённый каталог из двух источников

Status: done

## Story

As any user (authenticated or not),
I want to see approved user-submitted books alongside admin-curated books in the catalog,
so that the catalog reflects the collective expertise of the club's members.

## Acceptance Criteria

1. **[AC1 — Объединённый ответ]** GET `/api/books` возвращает объединённый список: книги из Google Sheets + заявки со статусом `approved` из `book_submissions`

2. **[AC2 — Только approved]** Заявки со статусом `pending` или `rejected` НЕ появляются в каталоге

3. **[AC3 — Публичный доступ]** GET `/api/books` не требует авторизации (auth не меняется — уже публичен)

4. **[AC4 — Нет ошибок без заявок]** Если одобренных заявок нет — возвращаются только Sheets-книги, без ошибок

5. **[AC5 — Формат совместим]** Одобренные заявки маппятся в формат `BookWithCover` (все обязательные поля присутствуют)

### Полная реализация `app/api/books/route.ts`

```typescript
// app/api/books/route.ts
export const dynamic = 'force-dynamic'

import { NextResponse } from 'next/server'
import { fetchBooks } from '@/lib/sheets'
import { db } from '@/lib/db'
import { bookStatuses, bookSubmissions } from '@/lib/db/schema'
import { eq } from 'drizzle-orm'

export async function GET() {
  try {
    const [books, statuses, approvedSubmissions] = await Promise.all([
      fetchBooks(),
      db.select().from(bookStatuses).catch(() => []),
      db.select().from(bookSubmissions).where(eq(bookSubmissions.status, 'approved')).catch(() => []),
    ])
    const statusMap = new Map(statuses.map(s => [s.bookId, s.status]))
    const booksWithStatus = books.map(b => ({ ...b, status: statusMap.get(b.id) ?? null }))
    const submissionBooks = approvedSubmissions.map(s => ({
      id: s.id,
      name: s.title,
      tags: s.topic ? [s.topic] : [],
      author: s.author,
      type: 'Book',
      size: '',
      pages: s.pages != null ? String(s.pages) : '',
      date: s.publishedDate ?? '',
      link: s.textUrl ?? '',
      description: s.description ?? '',
      coverUrl: s.coverUrl ?? null,
      status: null,
    }))
    return NextResponse.json({ books: [...booksWithStatus, ...submissionBooks] })
  } catch (e) {
    console.error('Failed to fetch books:', e)
    return NextResponse.json({ error: 'Failed to fetch books' }, { status: 500 })
  }
}
```

### Паттерн мока для тестов

Текущий мок в `route.test.ts` использует `db.select().from()`, но теперь нужно поддержать ДВА разных вызова:
1. `db.select().from(bookStatuses)` — возвращает Promise через `.catch()`
2. `db.select().from(bookSubmissions).where(eq(...))` — возвращает Promise через `.where().catch()`

Решение: `from()` возвращает объект, который **одновременно**:
- Имеет метод `.where()` → для запроса с фильтром
- Является thenable/PromiseLike → для запроса без фильтра (через `.catch()`)

```typescript
// Хелпер для создания chainable-результата Drizzle
function makeChainable(data: unknown[] = []) {
  const p = Promise.resolve(data)
  return {
    where: jest.fn(() => Promise.resolve(data)),
    then: p.then.bind(p),
    catch: p.catch.bind(p),
    finally: p.finally.bind(p),
  }
}
```

Структура мока в тестах:
```typescript
const mockSelect = jest.fn()

jest.mock('@/lib/db', () => ({
  db: { select: mockSelect },
}))
```

В каждом тесте настраивать через `mockSelect.mockReturnValue(...)`:
```typescript
beforeEach(() => {
  // Оба вызова select() → возвращают пустые данные по умолчанию
  mockSelect.mockReturnValue({
    from: jest.fn().mockReturnValue(makeChainable([])),
  })
})

// Для теста с approved submission:
it('возвращает approved заявки в списке книг', async () => {
  const mockApprovedSub = {
    id: 'sub-uuid-1', userId: 'user-1', title: 'Сапиенс', author: 'Харари',
    topic: 'История', pages: 500, publishedDate: '2011', textUrl: null,
    description: 'О человечестве', coverUrl: null,
    whyRead: 'Важно', status: 'approved',
    createdAt: new Date(), updatedAt: new Date(),
  }
  mockSelect.mockReturnValue({
    from: jest.fn().mockReturnValue(makeChainable([mockApprovedSub])),
  })
  // Первый from() вызов — bookStatuses, второй — bookSubmissions (with where)
  // Поскольку оба возвращают одинаковые данные, нужно mockReturnValueOnce:
})
```

**ВАЖНО:** т.к. `db.select()` вызывается дважды в `Promise.all`, нужно использовать `mockReturnValueOnce` для разграничения:

```typescript
// Первый select() — bookStatuses (пустой массив)
mockSelect.mockReturnValueOnce({
  from: jest.fn().mockReturnValue(makeChainable([])),
})
// Второй select() — bookSubmissions (approved)
mockSelect.mockReturnValueOnce({
  from: jest.fn().mockReturnValue(makeChainable([mockApprovedSub])),
})
```

**Примечание:** тест "не включает pending заявки" — на уровне unit-теста достаточно проверить, что в `where()` передаётся `eq(bookSubmissions.status, 'approved')`. Фактическая фильтрация в БД тестируется интеграционно.

### Важные детали

- **ID не конфликтуют:** Sheets-книги имеют числовые строковые ID ("1", "2", ...), submission IDs — UUID. Коллизия невозможна.
- **status: null** у submission-книг: поле `status` в `BookWithCover` означает reading-статус ('reading'|'read'), не статус заявки. Для submitted-книг всегда `null`.
- **Порядок:** approved-книги добавляются после Sheets-книг: `[...booksWithStatus, ...submissionBooks]`
- **`.catch(() => [])` guard:** оба запроса обёрнуты в `.catch()` → сбой БД не ломает Sheets-часть
