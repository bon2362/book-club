# Retrospective

Уроки, собранные по итогам реализованных проектов. Пополняется после каждой ретроспективы.

---

## 2026-02-23 — book-club (первый сеанс разработки)

### Проект
Next.js 14 сайт книжного клуба: Google Sheets как БД, NextAuth v5 (Google + email magic link), Neon Postgres, Vercel.

---

### Что съело больше всего лимитов

#### 1. Inline styles — постоянный налог на весь проект
Все компоненты написаны с inline style-объектами. BooksPage.tsx — 560+ строк, AdminPanel.tsx — 610+ строк. При каждом редактировании файл читался целиком.
- **Потери:** каждое чтение компонента в 3–4 раза дороже, чем если бы использовался Tailwind
- **Особенно болезненно** при внедрении тёмной темы: 5 компонентов пришлось читать и переписывать за один сеанс

**Правило на будущее:** использовать Tailwind (или CSS-модули) с первого дня. Темизация через `dark:` префиксы, а не через JS-переменные и замену хексов.

#### 2. Production-only баги — дорогие debugging-циклы
Три бага проявились только на проде, каждый потребовал цикл: деплой → проверка → гипотезы → фикс → деплой.
- **Trailing `\n` в GOOGLE_SHEETS_ID** — передавался через `input=value + '\n'` в subprocess
- **Статический рендеринг `/api/books`** — Next.js закэшировал маршрут без env vars на этапе build
- **MissingAdapter у Resend** — NextAuth v5 email-провайдеры требуют адаптер (известное ограничение)

**Правило на будущее:** перед первым деплоем сделать локальную проверку с реальными env vars. Добавить `console.log` для внешних ключей (SHEETS_ID, API keys) в dev-режиме. Знать заранее: NextAuth email → всегда нужен адаптер.

#### 3. Компонент написан трижды (AuthModal)
Сначала с Resend → убрали Resend (MissingAdapter) → добавили DrizzleAdapter и вернули Resend. Три полных версии одного файла.

**Правило на будущее:** спланировать auth-стек до написания кода. Если предполагается email-провайдер, сразу закладывать адаптер и схему БД.

#### 4. Плагины — overhead системных промптов
- `brainstorming` — оправдан для первого сеанса, полезен
- `frontend-design` — избыточен, если дизайн-направление задано текстом заранее
- `subagent-driven-development` — дорогой паттерн (implementer + spec reviewer + quality reviewer на каждую задачу). Окупается только на больших/командных проектах

**Правило на будущее:** для небольших проектов (< 15 компонентов) не использовать subagent-driven-development. frontend-design вызывать максимум один раз — для задания стилистических паттернов, а не для каждого компонента.

#### 5. Чтение файлов без плана правок
Паттерн: читаем файл → правим одно → снова читаем → правим другое. Каждое чтение большого файла дорого.

**Правило на будущее:** перед редактированием файла собрать все планируемые изменения в нём — и сделать их за один проход.

### Примерное распределение лимитов

| Тип работы | Доля |
|---|---|
| Чтение файлов перед правками | ~20–25% |
| Написание нового кода | ~25–30% |
| Production debugging | ~18–22% |
| Плагины и их overhead | ~10–15% |
| Реализация бэклога (9 фич) | ~10–15% |

Вывод: отладка обошлась примерно столько же, сколько написание первоначального кода.

---

### Что работало хорошо

- Разбивка на компоненты с чёткими границами — точечные правки без эффекта домино
- CSS-переменные для тёмной темы — одно место для значений, COLORS-константа как прослойка
- Inline скрипт в `<head>` для предотвращения flash при смене темы
- `effectiveUser = currentUser ?? savedUser` — простое решение для отслеживания сохранённого профиля без лишних re-fetch

### Итоговые правила для следующих проектов

1. **Tailwind с первого дня** — не inline styles
2. **Спланировать auth до написания кода** — какие провайдеры, нужен ли адаптер, нужна ли схема БД
3. **Проверить env vars локально** перед первым деплоем
4. **Собрать все правки к файлу** до его чтения — один проход на файл
5. **Не использовать subagent-driven-development** для проектов < 15 компонентов
6. **frontend-design** — максимум один вызов для задания стиля, не для каждого компонента
7. **Не встраивать секреты в тело bash-команд** — Claude Code сохраняет одобренные команды буквально в `~/.claude/projects/.../settings.local.json`. Скрипты вида `content.replace('KEY=', 'KEY=real_secret')` и `printf '%s' 'actual_key'` оставляют реальные значения в этом файле. Правило: передавать секреты через `cat file |` или читать из уже заполненного `.env.local`, не встраивать значения в текст команды.
