### Процессы разработки

#### 27. .prettierrc в репозитории · `XS`

Prettier настроен только через VSCode (`devcontainer.json`). Добавить `.prettierrc` в корень репо — чтобы форматирование было одинаковым у всех участников и в CI независимо от редактора.

#### 31. Husky + lint-staged · `S`

Pre-commit хук: запускать `eslint --fix` и `prettier --write` только на staged файлах. Не даёт коммитить явно сломанный или неотформатированный код.

#### 32. Валидация env-переменных · `S`

`process.env.X` возвращает `undefined` без предупреждений — ошибка проявляется глубоко в рантайме. Добавить `@t3-oss/env-nextjs` или zod-схему: при отсутствии обязательной переменной сервер падает на старте с понятным сообщением.

#### 34. Мониторинг ошибок (Sentry) · `S`

Продакшен-ошибки видны только если пользователь сообщит. Подключить Sentry (`@sentry/nextjs`) — бесплатный tier достаточен. Даёт трассировку, контекст запроса и алерты на email при необработанных ошибках.

### UI/UX


#### 57. Новые книги в начале списка · `XS`

Сейчас порядок книг в каталоге определяется порядком строк в Google Sheets. Добавить сортировку по дате добавления — новые книги показываются первыми. Для книг из заявок дата есть в `book_submissions.createdAt`; для книг из Sheets нужна колонка с датой (или сортировать по порядку строк в обратном направлении).

#### 22. Кнопка переключения тёмной темы · `M`

Тёмный режим работает через `prefers-color-scheme`, но пользователь не может переключить его вручную. Добавить кнопку в хедер (иконка солнца/луны), сохранять выбор в `localStorage`.

#### 23. Toast-уведомления об ошибках · `S`

Многие ошибки API молча проглатываются или меняют текст кнопки — пользователь может не понять, что что-то пошло не так. Добавить простой toast-компонент (fixed div снизу экрана) для отображения ошибок и успешных действий.

#### 24. Пагинация или виртуализация списка книг · `M`

Все книги загружаются сразу. При 50+ книгах на мобильном это заметно. Добавить простую пагинацию или `react-virtual` для виртуализации длинного списка.

### Функциональность

#### 41. Ранжированное голосование за книги · `L`

Пользователь в личном кабинете может расставить книги, на которые записался, по приоритету — от наиболее интересной к наименее. Администратор видит агрегированные приоритеты при разбивке по группам: кому какая книга важнее, чтобы оптимально распределить людей по читательским кругам. UI: drag-and-drop сортировка списка книг в личном кабинете.

#### 42. Разбивка участников по группам (admin) · `L`

Администратор выбирает, как разбить участников, записавшихся на одну книгу, по группам для обсуждения. UI в admin-панели: список книг → участники → ручное распределение по группам (drag-and-drop или выпадающий список). Результат сохраняется и, опционально, рассылается участникам группы по email.


#### 47. Саммари книги от участников · `L`

Авторизованный участник может написать короткое саммари (личные впечатления, выводы) по книге, которую клуб уже прочитал. Саммари привязано к книге и пользователю, отображается на странице книги или в карточке. Несколько участников могут оставить саммари по одной книге.

Затрагивает: новая таблица в `lib/db/schema.ts` (book_summaries: bookId, userId, text, createdAt), новый API-эндпоинт, UI на карточке/странице книги.


#### 53. Форма «Предложить книгу» не открывается автоматически после OAuth-редиректа · `M`

Флоу «Предложить книгу → авторизоваться через Google → вернуться» не открывает форму автоматически. Проверить, сохраняется ли `submitIntent` в `callbackUrl` после OAuth-редиректа. Вероятная причина: `callbackUrl` теряется при редиректе через accounts.google.com. Альтернатива: сохранять намерение в `localStorage` перед редиректом.

#### 56. Приём входящих писем на @slowreading.club · `S`

Сейчас письма уходят с `noreply@slowreading.club` — ответы пользователей никуда не доставляются. Настроить email forwarding: входящие на `noreply@slowreading.club` (или другой адрес типа `hello@slowreading.club`) пересылать на личный ящик. Варианты: Resend Inbound (если поддерживает), Cloudflare Email Routing (бесплатно, просто), Namecheap Email Forwarding. Требует добавления MX-записей в DNS домена slowreading.club.

#### 54. Личный кабинет пользователя · `M`

Отдельная страница (или секция в модалке) для авторизованного пользователя с двумя блоками:

- **Мои предложения** — список книг, которые пользователь предлагал: название, автор, текущий статус (на рассмотрении / одобрена / отклонена). Данные из `book_submissions` по `userId`. Опционально: кнопка отозвать заявку (если `pending`).
- **Мои контакты** — форма редактирования контактной информации (имя, Telegram): то, что сейчас спрашивается при первом входе в `ContactsForm`. Позволяет пользователю обновить данные в любой момент без обращения к администратору.

## Выполнено

### 53. Форма «Предложить книгу» открывается автоматически после OAuth-редиректа

`submitIntent` сохраняется в `localStorage` перед редиректом через Google OAuth. При маунте `BooksPage` проверяет ключ — если есть, открывает форму и удаляет ключ.

### 52. Ярлык «Новая» + фильтр по новинкам

Новая таблица `book_new_flags` в БД. Для книг из заявок — авто-флаг 30 дней с даты создания, если не переопределено. Для книг из Google Sheets — только ручная пометка. Вкладка «Новинки» в admin-панели с тумблером. Badge «Новая» на карточках и в строках таблицы. Кнопка-фильтр «Новинки» в каталоге.

### 51. Фильтр «Мои книги»

Кнопка «Мои книги» в строке фильтров для залогиненных пользователей с хотя бы одной записью. Фильтрует по `selectedBooks`, независим от фильтра «Показать прочитанные».

### 50. Кнопка «Предложить книгу» в табличном виде

Строка-кнопка «+ Предложить книгу» первой в таблице при табличном виде каталога.

### 49. Предзаполнение формы «Предложить книгу» из активных фильтров

При открытии формы передаются `filterTag` → поле «Тема» и `filterAuthor` → поле «Писатель».

### 48. Умная фильтрация по автору — поиск по вхождению

Поле `author` разбивается по `,` / ` и ` / `&` на отдельные имена. Фильтр по автору проверяет вхождение выбранного имени. `searchBooks` сделан generic — работает с любым типом, расширяющим `{ id, name, author }`.

### 46. Поле «Почему стоит прочитать» на карточке книги

Отображается при раскрытии карточки в стиле цитаты: вертикальная оранжевая черта слева, тёплый фон, курсивный текст. Работает для книг из заявок (`whyRead`) и из Google Sheets (`whyForClub`).

### 45. Колонка «Why for club» в Google Sheets

Колонка M (индекс 12) считывается в `lib/sheets.ts` как `whyForClub`, передаётся в `BookWithCover.whyRead`.

### 40. Залипающая кнопка сабмита в форме «Предложить книгу»

Форма переработана: заголовок и кнопка «Отправить» зафиксированы, область с полями прокручивается независимо.

### 38. Прочитанные книги: запись открыта, но скрыты по умолчанию

Переключатель «Показать прочитанные» в фильтрах. Книги со статусом «read» скрыты по умолчанию, но запись на них разрешена — кнопка «Хочу читать» активна.

### 44. Удаление заявки на книгу администратором

Кнопка «Удалить» в раскрытой строке заявки на вкладке «Заявки» admin-панели. Inline-подтверждение «Удалить навсегда? / Да, удалить / Отмена». `DELETE /api/admin/submissions/[id]` удаляет запись из `book_submissions`. Книги из Google Sheets администратор убирает вручную из таблицы.

`/api/admin/status` параллельно запрашивает GitHub Actions API и Vercel API. `AdminStatusBar` — client component с polling каждые 60с, цветные точки статуса. Вставлен в подвал `/admin` под статичными данными из #12.

### 8. Загрузка книг пользователями

Участники могут предлагать книги через форму «Предложить книгу». Заявки попадают в admin-панель на модерацию.

### 43. Причина отказа заявки на книгу

Текстовое поле в admin-панели при отклонении заявки. Причина сохраняется в `book_submissions.rejection_reason` и отправляется пользователю в email.

### 39. Убрать кнопку «Предложить книгу» из шапки на мобильных

Тесты запускаются в CI на каждый пуш в main. Google Sheets и upsertSignup замоканы через `NEXTAUTH_TEST_MODE=true` — тесты не зависят от внешних сервисов.

### 14. Удаление пользователя администратором

Кнопка «Удалить» в таблице участников с диалогом подтверждения. Удаляет пользователя из Postgres (каскадно accounts/sessions), помечает строку в Sheets G=TO DELETE + H=yes (is_deleted_by_admin). JWT callback проверяет существование пользователя в DB — при отсутствии возвращает null, что очищает cookie и разлогинивает пользователя на следующем запросе.
